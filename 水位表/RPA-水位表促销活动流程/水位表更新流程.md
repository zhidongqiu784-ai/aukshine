# 水位表更新流程

## 1. 流程目的

本流程用于更新水位表整体数据，包括分店铺基准销量、加权销量、库存推演、合计行生成、合计行 BD（预测）和最终预估销量。

核心目标：

- 基于店铺行数据计算分店铺基准销量和加权销量。
- 递推分店铺未来库存。
- 汇总生成或更新 `shop = '合计'` 的合计行。
- 只在合计行生成 `BD（预测）`。
- 预测完成后重新推演合计行库存。
- 最后统一刷新预估销量。

执行前提：

- 必须先执行《水位表更新日类型流程》。
- 因为本流程依赖店铺行已经更新好的 `type` 和 `coefficient`。

## 2. 涉及数据表

| 表名 | 作用 |
|---|---|
| `daily_sales` | 水位表主表，包含店铺行和合计行 |
| `inventory_base` | 提供真实库存汇总来源 |
| `sales_coefficient` | 提供 BD、基础类型等日类型系数 |
| `datetypetime` | 在合计行 BD（预测）时判断是否命中叠加基础类型 |

说明：

- 本流程不会往 `deal_date` 写入 `BD（预测）`。
- `BD（预测）` 只写在 `daily_sales` 的合计行。

## 3. 总体执行顺序

| 顺序 | 步骤 | 主要动作 | 修改范围 |
|---|---|---|---|
| 1 | 更新分店铺基准销量 | 用分店铺销量和最大系数计算 `base_sales` | 店铺行 |
| 2 | 更新分店铺加权基准销量 | 用最近历史基准销量计算未来 `weighted_sales` | 店铺行 |
| 3 | 分店铺库存推演 | 递推未来库存、可售天数、销售口径库存和在途 | 店铺行 |
| 4 | 生成或更新合计行 | 汇总分店铺数据，并按日期边界保护历史字段 | 合计行 |
| 4-1 | 合计行 BD 预测 | 基于合计行真实 BD 时间线生成 `BD（预测）` | 合计行 |
| 5 | 修正合计行库存 | 预测销量更新后重新递推合计库存 | 合计行 |
| 6 | 计算历史加权基准销量 | 按每条数据自身日期回看过去 30 天 | 全部行 |
| 7 | 更新预估销量 | 统一刷新 `maybe_sales = weighted_sales * coefficient` | 全部行 |

## 4. 步骤1：更新分店铺基准销量

按同一 `asin + country + date` 汇总分店铺销量，计算每个店铺行的 `base_sales`。

计算口径：

```text
base_sales = SUM(分店铺 sales) / MAX(分店铺 coefficient)
```

说明：

- 只处理 `shop != '合计'` 的店铺行。
- 处理范围为最近 30 天之后的数据。
- 这里依赖前置流程已经把店铺行系数更新好。

## 5. 步骤2：更新分店铺加权基准销量

基于最近历史基准销量，计算今天及未来店铺行的 `weighted_sales`。

历史取数范围：

- `CURDATE() - 30 天`
- 到 `CURDATE() - 1 天`

加权规则：

- 历史天数越少，越接近普通平均。
- 历史天数越多，越偏重最近数据。
- 最终结果保留 1 位小数。

说明：

- 只写入今天及未来店铺行。
- 这是后续预估销量和库存推演的基础。

## 6. 步骤3：分店铺库存推演

从今天开始，按店铺维度递推未来库存。

主要更新字段：

- `inventory`
- `days_for_sale`
- `sale_inventory`
- `estimate_days_for_sales`
- `on_the_way`

库存递推口径：

```text
今日库存结余 = 昨日库存结余 - 今日预估销量 + 今日补货
```

特殊处理：

- 如果库存扣减后为负，并且当天有补货，则从当天补货重新起算。
- 普通库存和销售口径库存会分别递推。
- 在途 `on_the_way` 会按未来补货合计重新计算。

## 7. 步骤4：生成或更新合计行

将同一 `asin + country + date` 下的店铺行汇总成 `shop = '合计'` 行。

合计行主要规则：

- 销量、补货、在途、预估销量等字段按店铺行汇总。
- 库存优先使用 `inventory_base` 中的真实库存汇总。
- 合计行 `coefficient` 使用店铺行中的最大系数。
- 合计行 `type` 使用最大系数对应的店铺行日类型。
- 合计行 `base_sales = SUM(分店铺 sales) / MAX(分店铺 coefficient)`。
- 今天及未来的 `weighted_sales` 使用昨天向前 30 天的合计基准销量计算；历史较完整时，最近 7 天、8～15 天、16～30 天权重分别为 50%、30%、20%。
- 已存在的合计行更新，不存在的合计行插入。

已存在合计行的日期保护规则：

- 昨天只允许更新 `sales` 和 `base_sales`，用于接收领星延迟同步的昨日订单。
- 今天及未来允许更新型号、系数、类型、库存、预估销量、补货、在途及加权基准销量等合计字段。
- 前天及更早日期不由步骤4改写上述已有合计字段。
- 昨天及更早的已有 `weighted_sales` 在步骤4中保持原值。

说明：

- 这一步完成后，合计行具备完整时间线。
- 后续合计行 BD（预测）基于这条时间线推演。
- 步骤6仍会单独重算最近历史区间的 `weighted_sales`；完整 Python 流程运行后的历史加权值以步骤6结果为准。

## 8. 步骤4-1：合计行 BD（预测）

本步骤只在合计行生成 BD（预测）。

生成位置：

- 只写入 `daily_sales`
- 只写入 `shop = '合计'`

不会写入：

- `deal_date`
- 店铺行 `daily_sales`

### 8.1 真实 BD 识别规则

预测起点只看合计行中的真实 BD。

识别真实 BD 时使用“包含 BD，但排除预测”的规则。

会被当作真实 BD：

- `BD`
- `BD (7月PD)`
- `BD (10月秋促)`
- `BD (黑五网一)`
- `淡季、BD`
- `淡季、BD (7月PD)` 等包含 `、BD` 的类型
- `旺季、2026世界杯、BD (7月PD)` 等多段复合真实 BD 类型

不会被当作真实 BD：

- `BD（预测）`
- `淡季、BD（预测）`
- 其他包含 `预测` 的类型

说明：

- 这样可以让 `淡季、BD` 也参与下一轮预测起点。
- `BD (7月PD)` 可以识别为真实 BD 活动，但大促类型不参与基础叠加，只代表最终活动类型。
- 同时避免把之前生成的 `BD（预测）` 当成真实活动，造成预测循环。

### 8.2 预测起点

系统按 `asin + country` 分组，找到最近一轮真实 BD。

预测开始日：

| 站点 | 国家 | 下一轮开始日 |
|---|---|---|
| 欧洲 | `DE / FR / ES / IT / UK` | 上一轮 BD 结束日 + 8 天 |
| 非欧洲 | `US / CA / JP` | 上一轮 BD 结束日 + 22 天 |

说明：

- 欧洲相当于完整空出 7 天。
- 非欧洲相当于完整空出 21 天。

### 8.3 预测周期

每轮 BD（预测）固定 14 天：

- 开始日 = 推演出来的预测开始日。
- 结束日 = 开始日 + 13 天。
- 只生成今天到未来 180 天内的预测。

### 8.4 阻断规则

如果预测出来的 14 天范围内，已经存在真实活动或专享活动，则整轮 BD（预测）不生成。

阻断类型包括：

- 真实 BD
- 真实 LD
- 大促 BD / LD
- 含叠加基础类型的 BD / LD
- 专享类型
- `7月PD`
- `10月秋促`
- `黑五网一`

说明：

- 判断的是整段 14 天范围。
- 只要预测周期内撞到真实活动或专享活动，这一轮预测就跳过。
- 已包含 `预测` 的类型不会参与阻断判断。

### 8.5 写入规则

BD（预测）只覆盖低优先级类型。

不会覆盖：

- 真实 BD / LD
- 大促 BD / LD
- 叠加真实 BD / LD
- 专享类型
- `7月PD`
- `10月秋促`
- `黑五网一`

普通基础日期写入：

```text
BD（预测）
```

如果预测日期命中 `datetypetime` 的 `叠加基础类型`，则写入：

```text
基础类型、BD（预测）
```

基础叠加规则：

- 只读取 `datetypetime.daytype_category` 为 `基础类型` 和 `叠加基础类型` 的基础层。
- 允许 `基础类型 + 叠加基础类型`，也允许多个 `叠加基础类型` 相互叠加。
- `BD（预测）` 是活动预测类型，只追加在最后，不参与基础叠加。
- 大促 BD / LD、专享类型不参与基础叠加。

示例：

| 原基础日类型 | 预测写入结果 |
|---|---|
| `日常` | `BD（预测）` |
| `旺季` | `BD（预测）` |
| `淡季` | `淡季、BD（预测）` |
| `旺季、2026世界杯` | `旺季、2026世界杯、BD（预测）` |

### 8.6 系数规则

普通预测：

```text
BD（预测）系数 = BD系数
```

叠加预测：

```text
淡季、BD（预测）系数 = 淡季系数 × BD系数
旺季、2026世界杯、BD（预测）系数 = 旺季系数 × 2026世界杯系数 × BD系数
```

说明：

- 普通预测直接读取 `sales_coefficient.type = 'BD'`。
- 叠加预测先拆出所有基础组成部分，再乘以 BD 系数。

### 8.7 预估销量

预测日类型和系数写入后，立即更新预测行：

```text
maybe_sales = weighted_sales × coefficient
```

适用类型：

- `BD（预测）`
- `淡季、BD（预测）`、`旺季、2026世界杯、BD（预测）` 等复合预测类型

## 9. 步骤5：修正合计行库存

合计行 BD（预测）会改变未来 `coefficient` 和 `maybe_sales`。

因此步骤5必须在步骤4-1之后执行，用最新预测销量重新递推合计行库存。

主要更新字段：

- `inventory`
- `days_for_sale`
- `sale_inventory`
- `estimate_days_for_sales`

说明：

- 如果不重新推演，BD（预测）带来的销量变化不会反映到合计库存中。

## 10. 步骤6：计算历史加权基准销量

以每条数据自己的 `date` 作为目标日期，回看目标日期前 30 天到前 1 天，计算历史加权基准销量。

说明：

- 用于补齐历史区间的 `weighted_sales`。
- 该步骤和未来预测推演分开处理。
- 该步骤在步骤4之后执行，因此会覆盖步骤4对最近历史 `weighted_sales` 的保留结果。

## 11. 步骤7：更新预估销量

最后统一刷新：

```text
maybe_sales = weighted_sales × coefficient
```

说明：

- 这是最终收尾步骤。
- 确保所有日类型、系数、加权销量变化后，预估销量保持同步。

## 12. 合计行日类型优先级

| 优先级 | 类型 | 说明 |
|---|---|---|
| 1 | 真实 BD / LD | 最高优先级 |
| 2 | 大促 BD / LD | 真实活动在大促期间的修正类型 |
| 3 | 叠加真实 BD / LD | 例如 `淡季、BD` |
| 4 | 专享类型 | 例如 `7月PD(专享)` |
| 5 | BD（预测） | 只补充没有真实活动和专享的未来日期 |
| 6 | 叠加 BD（预测） | 例如 `淡季、BD（预测）` |
| 7 | 基础类型 | 日常、淡季、旺季等 |

说明：

- `BD（预测）` 不覆盖真实活动。
- `BD（预测）` 不覆盖专享。
- 叠加基础类型会影响预测类型和预测系数。
- 基础叠加层先形成 `旺季、2026世界杯` 这类组合，再追加真实或预测活动。
- 大促 BD / LD 只代表最终活动类型，不参与基础叠加。

## 13. 验证清单

| 验证项 | 判断标准 |
|---|---|
| 执行顺序 | 先执行日类型流程，再执行水位表更新流程 |
| 分店铺基准销量 | 店铺行 `base_sales` 按销量和最大系数更新 |
| 分店铺加权销量 | 今天及未来店铺行 `weighted_sales` 有值 |
| 分店铺库存 | 店铺行未来库存、可售天数、在途被递推 |
| 合计行生成 | `shop='合计'` 行存在并按国家、ASIN、日期汇总 |
| 昨日销量回补 | 昨日合计行只更新 `sales`、`base_sales`，其他合计字段保持原值 |
| 历史字段保护 | 步骤4不改写前天及更早的已有合计字段 |
| 合计行日类型 | 合计行按分店铺最大系数确定日类型 |
| 真实 BD 识别 | `BD`、`BD (7月PD)`、`淡季、BD`、`旺季、2026世界杯、BD (7月PD)` 都能作为真实 BD |
| 排除预测 | `BD（预测）` 不作为真实 BD 起点 |
| BD 预测位置 | `BD（预测）` 只出现在合计行 |
| BD 预测间隔 | 欧洲空 7 天，非欧洲空 21 天 |
| BD 预测周期 | 每轮 14 天 |
| 阻断规则 | 预测 14 天范围内有真实活动或专享时整轮跳过 |
| 基础叠加范围 | 只有基础类型和叠加基础类型参与基础叠加 |
| 叠加预测 | 旺季和 2026世界杯预测日期写成 `旺季、2026世界杯、BD（预测）` |
| 普通预测系数 | `BD（预测）` 使用 BD 系数 |
| 叠加预测系数 | `旺季、2026世界杯、BD（预测）` 使用旺季系数 × 2026世界杯系数 × BD系数 |
| 合计库存重算 | 预测销量变化后合计行库存重新递推 |
| 最终预估销量 | `maybe_sales = weighted_sales × coefficient` |
