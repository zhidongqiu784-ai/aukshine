# 水位表更新日类型流程

## 1. 流程目的

本流程用于每天更新店铺行 `daily_sales` 的日类型和系数。

核心目标：

- 先从 `datetypetime` 恢复基础日类型。
- 再从 `deal_date` 写入真实 BD / LD。
- 再根据 `datetypetime` 修正大促 BD / LD 和专享类型。
- 最后按最终日类型更新 `coefficient`。

核心原则：

- 只更新今天到未来 180 天。
- 昨天及以前的历史日类型和系数不自动覆盖。
- 取消 BD / LD 后，只影响今天及未来，不反向洗掉历史活动口径。

说明：

- 本流程只处理店铺行，即 `shop != '合计'`。
- 本流程不生成 `BD（预测）`。
- `BD（预测）` 只在《水位表更新流程》的合计行步骤中生成。

## 2. 涉及数据表

| 表名 | 作用 |
|---|---|
| `daily_sales` | 店铺行每日数据，写入最终日类型和系数 |
| `datetypetime` | 维护基础日类型、叠加基础类型、大促 BDLD、专享类型的日期区间 |
| `deal_date` | 提供真实 BD / LD 活动 |
| `sales_coefficient` | 提供不同日类型对应的系数 |

## 3. 总体执行顺序

| 顺序 | 步骤 | 主要动作 | 修改表 |
|---|---|---|---|
| 1 | 恢复默认类型 | 从 `datetypetime` 读取基础类型和叠加基础类型，先恢复基础日类型 | `daily_sales` |
| 2 | 更新真实 BD/LD | 从 `deal_date` 读取状态有效的真实 BD / LD | `daily_sales` |
| 3 | 修正大促和专享 | 从 `datetypetime` 读取大促 BDLD 和专享类型，进一步修正日类型 | `daily_sales` |
| 4 | 更新系数 | 根据最终日类型匹配或计算系数 | `daily_sales` |

执行范围统一为：

```sql
CURDATE() 到 DATE_ADD(CURDATE(), INTERVAL 180 DAY)
```

## 4. 步骤1：恢复基础日类型

步骤1先给今天到未来 180 天的店铺行恢复基础日类型。

读取 `datetypetime` 的规则：

- `daily_sales.date` 必须落在 `datetypetime.startdate` 到 `datetypetime.enddate` 之间。
- 国家按区域匹配：
  - `datetypetime.country` 存储无空格的国家代码列表，例如 `US,CA`、`UK,DE,FR,ES,IT`、`JP`
  - SQL 通过 `FIND_IN_SET(业务国家代码, datetypetime.country) > 0` 匹配区域配置
  - 后续调整国家归组时只修改 `datetypetime.country`，不再修改流程 SQL
- 只读取 `daytype_category` 为以下两类的数据：
  - `基础类型`
  - `叠加基础类型`

写入规则：

- 匹配到 `datetypetime.daytype`，就写入对应日类型。
- 如果同一天同时命中基础类型和叠加基础类型，则按 `、` 拼成基础组合。
- 没有匹配到配置，就写入 `日常`。

基础叠加规则：

- 只有 `基础类型` 和 `叠加基础类型` 参与基础日类型叠加。
- 允许 `基础类型 + 叠加基础类型`，也允许多个 `叠加基础类型` 相互叠加。
- `BD`、`LD`、`BD (7月PD)`、`LD (7月PD)`、专享类型不参与基础叠加。

示例：

| 日期配置 | 分类 | 写入结果 |
|---|---|---|
| 旺季区间 | 基础类型 | `旺季` |
| 淡季区间 | 叠加基础类型 | `淡季` |
| 旺季区间 + 2026世界杯区间 | 基础类型 + 叠加基础类型 | `旺季、2026世界杯` |
| 没有配置 | 无 | `日常` |

说明：

- `基础类型` 是当天的基础日类型。
- `叠加基础类型` 可以先和基础类型组合，再和 BD / LD 组合，例如后续变成 `旺季、2026世界杯、BD`。

## 5. 步骤2：更新真实 BD / LD

步骤2从 `deal_date` 读取真实活动，并覆盖店铺行日类型。

匹配规则：

- `daily_sales.country = deal_date.country`
- `daily_sales.shop = deal_date.shop_code`
- `daily_sales.asin = deal_date.asin`
- `deal_date.promotion_type` 必须是 `BD` 或 `LD`
- `daily_sales.date` 必须在活动开始时间和结束时间内
- `deal_date.origin_status` 必须是有效状态

有效状态：

- `进行中`
- `未开始`
- `已结束`
- `未定`
- `未定义类型`

不参与写入：

- `已取消`
- `待申报`
- `需要注意`
- 其他未列入有效状态的值

写入规则：

| 当前基础日类型 | 活动类型 | 写入结果 |
|---|---|---|
| `日常` | BD | `BD` |
| `旺季` | BD | `BD` |
| `淡季` | BD | `淡季、BD` |
| `淡季` | LD | `淡季、LD` |

说明：

- 普通基础类型遇到真实 BD / LD，会直接写成 `BD` 或 `LD`。
- 叠加基础类型遇到真实 BD / LD，会保留基础影响，写成 `基础类型、BD` 或 `基础类型、LD`。

## 6. 步骤3：修正大促 BD/LD 与专享类型

步骤3继续从 `datetypetime` 读取特殊配置。

读取 `datetypetime` 的规则：

- 日期和国家区域匹配规则与步骤1一致。
- 只读取 `daytype_category` 为以下两类的数据：
  - `大促BDLD`
  - `专享类型`

大促 BDLD 规则：

- 大促 BDLD 不参与基础叠加。
- 它只在已有真实 BD / LD 后，把最后的活动类型替换成更具体的大促活动类型。
- 普通 `BD` 在 7月PD 区间内，修正为 `BD (7月PD)`。
- `旺季、2026世界杯、BD` 在 7月PD 区间内，修正为 `旺季、2026世界杯、BD (7月PD)`。
- 不会写成 `旺季、2026世界杯、BD、BD (7月PD)`。

专享类型规则：

- 当前日类型不是 `BD` / `LD`，且不是 `淡季、BD` / `淡季、LD` 这类复合类型时，才允许写入专享类型。
- 这样可以避免专享覆盖真实 BD / LD 或叠加 BD / LD。

示例：

| 当前日类型 | 命中配置 | 最终结果 |
|---|---|---|
| `BD` | `BD (7月PD)` | `BD (7月PD)` |
| `LD` | `LD (黑五网一)` | `LD (黑五网一)` |
| `旺季、2026世界杯、BD` | `BD (7月PD)` | `旺季、2026世界杯、BD (7月PD)` |
| `日常` | `7月PD(专享)` | `7月PD(专享)` |
| `淡季、BD` | 专享类型 | 不覆盖，保留 `淡季、BD` |

## 7. 步骤4：更新系数

步骤4根据最终日类型更新 `daily_sales.coefficient`。

### 7.1 普通类型

普通类型直接用最终 `daily_sales.type` 匹配 `sales_coefficient.type`。

示例：

| 最终日类型 | 系数来源 |
|---|---|
| `日常` | `sales_coefficient.type = '日常'` |
| `BD` | `sales_coefficient.type = 'BD'` |
| `BD (7月PD)` | `sales_coefficient.type = 'BD (7月PD)'` |

### 7.2 复合类型

复合类型按 `、` 拆成多个组成部分，分别取系数后全部相乘。

示例：

| 最终日类型 | 计算方式 |
|---|---|
| `淡季、BD` | `淡季系数 × BD系数` |
| `淡季、LD` | `淡季系数 × LD系数` |
| `旺季、2026世界杯、BD` | `旺季系数 × 2026世界杯系数 × BD系数` |
| `旺季、2026世界杯、BD (7月PD)` | `旺季系数 × 2026世界杯系数 × BD (7月PD)系数` |

说明：

- 系数也只更新今天到未来 180 天。
- 历史系数不再被反向覆盖。
- 这里 `BD (7月PD)` 作为最终活动类型参与系数计算，但不作为基础叠加类型。

## 8. 取消 BD / LD 时的处理

如果 `deal_date` 中某个 BD / LD 后续变成 `已取消`：

- 昨天及以前：不再自动修改，保留历史日类型和系数。
- 今天及未来：按最新状态重算，已取消活动不再写入 BD / LD。
- 系数同步按最新日类型更新。

示例：

| 日期范围 | 处理结果 |
|---|---|
| 活动取消前已经发生的日期 | 历史冻结，不自动改 |
| 今天到未来活动区间 | 因状态已取消，不再写 BD / LD |
| 未来普通日期 | 恢复为基础日类型 |

## 9. 店铺行日类型优先级

| 优先级 | 类型 | 来源 |
|---|---|---|
| 1 | 大促 BD / LD | `datetypetime.daytype_category = '大促BDLD'` |
| 2 | 真实 BD / LD | `deal_date` |
| 3 | 叠加真实 BD / LD | 基础类型 + `deal_date`，例如 `淡季、BD` |
| 4 | 专享类型 | `datetypetime.daytype_category = '专享类型'` |
| 5 | 基础类型 | `基础类型`、`叠加基础类型`、默认 `日常` |

说明：

- 基础叠加层先确定基础日类型组合，例如 `旺季、2026世界杯`。
- 活动层再追加或替换最后的 BD / LD，例如 `旺季、2026世界杯、BD (7月PD)`。
- 大促 BD / LD 只修正活动层，不参与基础叠加层。
- 店铺行不生成 `BD（预测）`。
- 合计行预测由《水位表更新流程》处理。

## 10. 与水位表更新流程的关系

完整执行顺序应为：

1. 先执行《水位表更新日类型流程》对应的 Python。
2. 再执行《水位表更新流程》对应的 Python。

原因：

- 第一个流程先把店铺行日类型和系数准备好。
- 第二个流程再基于店铺行汇总合计行、推演库存和生成合计行 BD（预测）。

## 11. 验证清单

| 验证项 | 判断标准 |
|---|---|
| 更新范围 | 只更新今天到未来 180 天 |
| 历史保护 | 昨天及以前的 `type` 和 `coefficient` 不被自动覆盖 |
| 基础类型 | 普通日期能恢复为 `日常`、`旺季` 等基础类型 |
| 叠加基础类型 | 淡季日期先恢复为 `淡季` |
| 基础叠加范围 | 只有基础类型和叠加基础类型参与基础叠加 |
| 多重基础叠加 | 旺季和 2026世界杯同一天写成 `旺季、2026世界杯` |
| 真实 BD/LD | 有效状态的真实活动能写入 `BD` / `LD` |
| 取消活动 | `已取消`、`待申报` 不写入 BD / LD |
| 叠加活动 | 淡季遇到 BD 能写成 `淡季、BD` |
| 大促修正 | `旺季、2026世界杯、BD` 能修正为 `旺季、2026世界杯、BD (7月PD)` |
| 专享保护 | 专享不覆盖 `淡季、BD` 等复合类型 |
| 普通系数 | 普通日类型能直接匹配对应系数 |
| 复合系数 | `旺季、2026世界杯、BD` 能按全部组成部分相乘 |
