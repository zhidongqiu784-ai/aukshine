# 库存风险 & 安全天数趋势

## 1. 图表定位

该图表用于查看单个 ASIN 在指定国家和店铺维度下，未来 180 天内的库存安全天数变化、补货节点分布以及低于安全线的风险时间点。

核心关注点：

- 什么时候销售预估安全天数低于安全线
- 什么时候系统预估安全天数低于安全线
- 哪些日期存在补货节点
- 每个补货节点对应哪些补货单
- 当前库存、销售库存、在途数量和日均预估是否支撑未来销售

当前安全线为：

- 目标 7 天
- 目标 14 天

## 2. 页面参数

图表根据当前 URL 参数读取数据：

- `asin`：商品 ASIN
- `country`：国家
- `shop`：店铺

店铺逻辑：

- 当 `shop = 合计` 时，补货数据只按 `asin` 和 `country` 查询
- 当 `shop != 合计` 时，补货数据按 `asin`、`country`、`shop` 查询

这样可以保证合计视图看到全部店铺补货，单店铺视图只看当前店铺补货。

## 3. 主数据来源

### daily_sales

主曲线数据来自 `daily_sales`。

查询范围：

- 起始日期：当天
- 结束日期：当天之后 180 天

主要使用字段：

- `date`：图表日期
- `days_for_sale`：系统预估安全天数
- `estimate_days_for_sales`：销售预估安全天数
- `maybe_sales`：当天日均预估
- `inventory`：系统库存
- `sale_inventory`：销售库存
- `on_the_way`：在途数量
- `add`：原始补货字段，当前不作为最终补货节点数量来源

图表中的补货节点数量，当前按 `simulate_shipment.number` 汇总，不直接使用 `daily_sales.add`。

## 4. 补货数据来源

### simulate_shipment

补货记录来自 `simulate_shipment`。

查询条件：

- `asin = 当前 URL asin`
- `country = 当前 URL country`
- `add_date >= 今天`
- 如果 URL 店铺不是 `合计`，追加 `shop = 当前 URL shop`

主要使用字段：

- `add_date`：模拟补货到达日期
- `date`：模拟发货日期
- `number`：补货数量
- `shippment_id`：真实发货单号关联键
- `logistics_channel`：模拟物流渠道
- `channel`：兼容字段
- `shop`：店铺

### delivery_note

当 `simulate_shipment` 存在 `shippment_id` 时，使用该字段关联 `delivery_note.shipment_id` 查询真实发货信息。

主要使用字段：

- `shipment_id`：真实发货单号
- `shipment_time`：真实发货日期
- `logistics_channel_name`：真实物流渠道
- `estimated_arrival_date`：预计到达日期
- `expected_storage_time`：预计入库时间

## 5. 补货节点计算规则

补货节点不是简单按 `simulate_shipment.add_date` 展示，而是根据是否存在真实发货单决定。

### 有 shippment_id

如果 `simulate_shipment` 有 `shippment_id`，则查询 `delivery_note`。

补货节点日期优先级：

1. `delivery_note.estimated_arrival_date`
2. `delivery_note.expected_storage_time`

悬浮框展示：

- 发货日期：`delivery_note.shipment_time`
- 物流渠道：`delivery_note.logistics_channel_name`

### 无 shippment_id

如果 `simulate_shipment` 没有 `shippment_id`，则使用模拟发货数据。

补货节点日期：

- `simulate_shipment.add_date`

悬浮框展示：

- 发货日期：`simulate_shipment.date`
- 物流渠道：`simulate_shipment.logistics_channel`
- 兼容 `channel` 和 `logistics_channel.logistics_days`

### 同一天多张单

同一天存在多张补货单时：

- 图表节点数量按当天所有单的 `simulate_shipment.number` 汇总
- 悬浮框中按单据分列展示
- 不把多张单挤在一行文本中

示例：

- 2026-07-17 有两张单
- 每张单数量为 200
- 图表节点数量应显示 400

## 6. 曲线与节点展示

### 系统预估安全天数

- 来源字段：`daily_sales.days_for_sale`
- 展示为蓝色曲线
- 曲线下方有浅蓝色面积
- 用于观察系统口径下库存可售天数

### 销售预估安全天数

- 来源字段：`daily_sales.estimate_days_for_sales`
- 展示为黄色曲线
- 用于观察销售口径下库存可售天数

### 补货节点

- 来源：按规则汇总后的补货日期和补货数量
- 节点显示为圆点
- 节点旁常驻显示补货数量
- 数量只显示数值，不显示“补”字

## 7. 顶部风险摘要

图表顶部展示三个摘要信息：

- 首次低于 14 天
- 首次低于 7 天
- 补货节点数量与补货总数

首次低于安全线的判断基于销售预估安全天数。

摘要中的安全线文本与图表安全线保持一致，当前为 7 天和 14 天。

## 8. 悬浮框内容

鼠标滑动到图表节点时，展示表格样式悬浮框。

基础信息包括：

- 系统预估安全天数
- 销售预估安全天数
- 补货数量
- 当天日均预估
- 系统库存
- 销售库存
- 在途数量

如果当天存在补货单，额外展示补货明细表。

补货明细表包括：

- 单号
- 店铺
- 数量
- 发货日
- 渠道

## 9. 当前实现形式

当前 [安全库存图表.js](../水位表详情/安全库存图表.js) 使用 NocoBase JS Block 渲染。

实现特点：

- 使用 React 组件渲染图表
- 使用 SVG 绘制曲线、面积、节点和安全线
- 使用 NocoBase API 查询 `daily_sales`、`simulate_shipment`、`delivery_note`
- 不使用 `document`、`window` 或 DOM 操作
- 不通过字符串拼接注入页面结构

## 10. 验证重点

后续验证该图表时，建议按以下路径检查。

### 店铺切换

- 打开水位表详情页
- 切换店铺按钮
- 确认 URL 中 `shop` 参数变化
- 确认图表标题、曲线、补货节点与当前店铺一致

### 合计店铺

- URL 中 `shop=合计`
- 确认补货记录不按店铺过滤
- 确认多个店铺补货单可以汇总到同一天节点

### 单店铺

- URL 中 `shop` 为具体店铺
- 确认补货记录只显示当前店铺数据

### 补货节点

- 找到存在多张单的补货日期
- 确认节点数量等于当天所有 `simulate_shipment.number` 之和
- 鼠标悬浮时确认补货明细分列展示

### 真实发货单

- 找到存在 `shippment_id` 的模拟发货记录
- 确认节点日期使用 `delivery_note.estimated_arrival_date`
- 如果该字段为空，确认使用 `delivery_note.expected_storage_time`
- 确认悬浮框展示真实发货日期和真实物流渠道

### 模拟发货单

- 找到没有 `shippment_id` 的模拟发货记录
- 确认节点日期使用 `simulate_shipment.add_date`
- 确认悬浮框展示模拟发货日期和模拟物流渠道

### 安全线

- 确认图表安全线为 7 天和 14 天
- 确认顶部摘要同步显示首次低于 7 天和首次低于 14 天

## 11. 注意事项

- `shippment_id` 字段拼写沿用当前表结构，不额外改名
- 补货节点数量以 `simulate_shipment.number` 为准
- `daily_sales.add` 当前不作为节点数量汇总依据
- 如果真实发货单没有匹配到 `delivery_note`，该真实单不应强行用错误日期展示
- 如果某天没有补货单，悬浮框只展示基础库存与安全天数信息
