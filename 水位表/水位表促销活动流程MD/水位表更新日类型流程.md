# 水位表更新日类型流程

## 1. 流程目的

用于说明店铺行 `daily_sales.type` 和 `daily_sales.coefficient` 的刷新顺序。

核心目标：

- 先把店铺行恢复为基础日类型。
- 再用真实 BD / LD 覆盖店铺行日类型。
- 再修正大促 BD / LD 和专享日类型。
- 最后按最终日类型更新系数。

说明：本模块不再生成 `BD（预测）`，`BD（预测）` 只在“水位表更新流程”的合计行步骤中生成。

## 2. 涉及数据表

| 表名 | 作用 |
|---|---|
| `deal_date` | 提供真实 BD / LD 活动来源；旧的 `BD（预测）` 会被清理 |
| `daily_sales` | 店铺行每日数据，最终写入日类型和系数 |
| `datetypetime` | 存放日常、淡季、淡季过渡、专享、大促 BD/LD 日期区间 |
| `sales_coefficient` | 存放不同日类型对应的系数 |

## 3. 总体执行顺序

| 顺序 | 步骤 | 主要动作 | 修改表 |
|---|---|---|---|
| 0 | 删除旧 BD 预测 | 清理历史遗留的 `BD（预测）`，避免旧预测继续影响页面或后续流程 | `deal_date` |
| 1 | 恢复默认类型 | 先恢复成日常 / 淡季 / 淡季过渡 / 专享 | `daily_sales` |
| 2 | 更新真实 BD/LD | 用 `deal_date` 中真实活动覆盖店铺行日类型 | `daily_sales` |
| 3 | 修正大促和专享 | 大促期间真实 BD/LD 改成大促类型；无真实 BD/LD 时保留专享 | `daily_sales` |
| 4 | 更新系数 | 按店铺行最终日类型匹配系数 | `daily_sales` |

## 4. 各步骤说明

### 步骤0：删除旧 BD 预测

清理 `deal_date` 中历史遗留的预测记录：

- `origin_status = '预测'`
- `promotion_type = 'BD（预测）'`

说明：

- 新逻辑不再往 `deal_date` 生成 `BD（预测）`。
- 这一步只做历史清理，避免旧预测继续残留。
- 不删除真实 BD / LD。

### 步骤1：恢复默认类型

先把店铺行 `daily_sales.type` 恢复为基础状态。

写入范围：

- `日常`
- `淡季`
- `淡季过渡`
- `7月PD(专享)`
- `10月秋促(专享)`
- `黑五网一(专享)`

说明：

- 这一步相当于先清理上一轮活动覆盖结果。
- 后续再用真实 BD/LD 覆盖。
- 只处理 `shop != '合计'` 的店铺行。

### 步骤2：更新真实 BD / LD

根据 `deal_date` 中真实 BD/LD 覆盖店铺行 `daily_sales.type`。

参与类型：

- `BD`
- `LD`

参与状态：

- `进行中`
- `未开始`
- `已结束`
- `未定`
- `未定义类型`

不参与状态：

- `需要注意`

说明：

- 真实 BD/LD 优先级最高。
- 真实活动可以覆盖日常、淡季、淡季过渡和专享。
- 只处理 `shop != '合计'` 的店铺行。

### 步骤3：修正大促 BD/LD 与专享

根据 `datetypetime` 修正大促期间的最终类型。

真实活动修正规则：

| 原类型 | 大促区间 | 最终显示 |
|---|---|---|
| `BD` | 7月PD | `BD (7月PD)` |
| `LD` | 7月PD | `LD (7月PD)` |
| `BD` | 10月秋促 | `BD (10月秋促)` |
| `LD` | 10月秋促 | `LD (10月秋促)` |
| `BD` | 黑五网一 | `BD (黑五网一)` |
| `LD` | 黑五网一 | `LD (黑五网一)` |

专享规则：

- 没有真实 BD/LD 时，显示对应专享。
- 专享包括 `7月PD(专享)`、`10月秋促(专享)`、`黑五网一(专享)`。
- 专享不能覆盖真实 BD/LD。

### 步骤4：更新系数

根据店铺行最终 `daily_sales.type` 匹配 `sales_coefficient`。

说明：

- 普通类型按自身类型匹配系数。
- 本模块不处理 `BD（预测）` 的系数。
- 合计行 `BD（预测）` 的系数在“水位表更新流程”中使用 `BD` 系数。

## 5. 店铺行日类型优先级

| 优先级 | 类型 | 说明 |
|---|---|---|
| 1 | 真实 BD / LD | 来自 `deal_date` 的真实活动 |
| 2 | 大促 BD / LD | 真实 BD/LD 落在大促期间后的修正类型 |
| 3 | 专享 | 无真实 BD/LD 时显示专享 |
| 4 | 基础类型 | 日常、淡季、淡季过渡 |

说明：店铺行不生成 `BD（预测）`，所以店铺行优先级里不包含 `BD（预测）`。

## 6. 与合计行 BD 预测的关系

- 店铺行先完成日类型和系数刷新。
- 水位表更新流程再根据店铺行生成或更新 `shop='合计'` 行。
- `BD（预测）` 只在合计行上推演和显示。
- 店铺级预测不再生成，避免不同店铺预测时间不一致导致合计行连续出现预测 BD。

## 7. 维护注意事项

- 不再生成“待申报BD”。
- 不再按店铺生成 `BD（预测）`。
- `deal_date` 只保留真实活动来源，旧预测只做清理。
- `需要注意` 不参与写入 `daily_sales`。
- 专享高于合计行 `BD（预测）`，但店铺行本身不显示预测。

## 8. 验证清单

| 验证项 | 判断标准 |
|---|---|
| 旧预测清理 | `deal_date` 中历史 `BD（预测）` 被清理 |
| 默认类型恢复 | 店铺行先恢复为日常、淡季、淡季过渡或专享 |
| 真实 BD/LD 覆盖 | 店铺行真实活动日期显示 `BD` 或 `LD` |
| 大促类型修正 | 大促日期内真实 BD/LD 显示为对应大促类型 |
| 专享保留 | 没有真实 BD/LD 时，大促专享日期显示专享 |
| 系数更新 | 店铺行系数与最终日类型匹配 |
| 不生成预测 | 店铺行不出现新生成的 `BD（预测）` |
