# 水位表更新流程

## 1. 流程目的

用于说明水位表整体更新顺序，包括分店铺销量、库存推演、合计行生成、合计行 BD 预测和最终预估销量刷新。

核心目标：

- 先更新分店铺基础数据。
- 再汇总生成 `shop='合计'` 行。
- 再只基于合计行推演 `BD（预测）`。
- 最后重新推演合计库存并刷新预估销量。

## 2. 涉及数据表

| 表名 | 作用 |
|---|---|
| `daily_sales` | 水位表主表，保存店铺行和合计行 |
| `inventory_base` | 提供真实库存汇总来源 |
| `sales_coefficient` | 提供 BD 等日类型系数 |

说明：本流程不会往 `deal_date` 插入 `BD（预测）`。

## 3. 总体执行顺序

| 顺序 | 步骤 | 主要动作 | 修改范围 |
|---|---|---|---|
| 1 | 更新分店铺基准销量 | 用分店铺销量和最大系数计算 `base_sales` | 店铺行 |
| 2 | 更新分店铺加权基准销量 | 用最近历史基准销量计算未来 `weighted_sales` | 店铺行 |
| 3 | 分店铺库存推演 | 递推未来库存、可售天数、销售口径库存和在途 | 店铺行 |
| 4 | 生成或更新合计行 | 汇总分店铺数据生成 `shop='合计'` 行 | 合计行 |
| 4-1 | 合计行 BD 预测 | 只基于合计行真实 BD 时间线生成 `BD（预测）` | 合计行 |
| 5 | 修正合计行库存 | 在预测日类型和销量更新后重新递推合计库存 | 合计行 |
| 6 | 计算历史加权基准销量 | 按每条数据自身日期回看过去 30 天 | 全部行 |
| 7 | 更新预估销量 | 统一刷新 `maybe_sales = weighted_sales * coefficient` | 全部行 |

## 4. 各步骤说明

### 步骤1：更新分店铺基准销量

按同一 `asin + country + date` 汇总分店铺销量。

计算口径：

- `base_sales = SUM(sales) / MAX(coefficient)`

说明：

- 只处理 `shop != '合计'` 的店铺行。
- 用于后续计算加权基准销量。

### 步骤2：更新分店铺加权基准销量

基于最近历史基准销量计算未来 `weighted_sales`。

说明：

- 取最近历史数据作为基础。
- 根据历史天数分段加权。
- 写入今天及未来店铺行。

### 步骤3：分店铺库存推演

从今天开始递推分店铺未来库存。

主要更新：

- `inventory`
- `days_for_sale`
- `sale_inventory`
- `estimate_days_for_sales`
- `on_the_way`

说明：

- 库存按“昨日结余 - 预估销量 + 补货”递推。
- 如果库存扣减为负且当天有补货，则以当天补货重新起算。
- 同时处理普通库存口径和销售口径库存。

### 步骤4：生成或更新合计行

将同一 `asin + country + date` 下的分店铺数据汇总成 `shop='合计'` 行。

合计行主要口径：

- 销量、补货、在途、预估销量等做汇总。
- 库存优先取 `inventory_base` 汇总库存。
- 合计行日类型按分店铺中最大系数对应的类型确定。
- 已有合计行会更新，缺失合计行会插入。

说明：

- 这一步结束后，合计行已经具备完整时间线。
- 后续 `BD（预测）` 基于合计行时间线推演。

## 5. 合计行 BD 预测生成逻辑

### 5.1 生成位置

`BD（预测）` 只写入 `daily_sales` 中的合计行：

- `shop = '合计'`

不写入：

- `deal_date`
- 店铺行 `daily_sales`

### 5.2 预测来源

只根据合计行中的真实 BD 时间线推演。

参与作为预测基准的类型：

- `BD`
- `BD (7月PD)`
- `BD (10月秋促)`
- `BD (黑五网一)`

说明：

- 按同一 `asin + country + shop='合计'` 分组。
- 取最近一轮真实 BD 的结束日期作为起点。

### 5.3 预测间隔

| 站点 | 国家 | 下一轮开始日 |
|---|---|---|
| 欧洲 | `DE / FR / ES / IT / UK` | 上一轮结束日 + 8 天 |
| 非欧洲 | `US / CA / JP` | 上一轮结束日 + 22 天 |

说明：

- 欧洲是完整空出 7 天，所以 SQL 使用 `+ 8 天`。
- 非欧洲是完整空出 21 天，所以 SQL 使用 `+ 22 天`。

### 5.4 预测周期

每轮 `BD（预测）` 固定 14 天：

- 开始日 = 推演出来的开始日期。
- 结束日 = 开始日 + 13 天。
- 只生成今天及未来 180 天内的日期。

### 5.5 阻断规则

判断的是预测出来的 14 天范围内是否已有更高优先级日类型。

如果预测 14 天范围内存在以下任意类型，则这一整轮 `BD（预测）` 不生成：

- `BD`
- `LD`
- `BD (7月PD)`
- `LD (7月PD)`
- `BD (10月秋促)`
- `LD (10月秋促)`
- `BD (黑五网一)`
- `LD (黑五网一)`
- `7月PD(专享)`
- `10月秋促(专享)`
- `黑五网一(专享)`

说明：

- 是判断预测整段 14 天范围，不是只判断预测开始日前后。
- 只要预测段内碰到真实活动或专享，这一轮整段跳过。

### 5.6 写入规则

只覆盖基础类型，不覆盖高优先级类型。

不会覆盖：

- 真实 BD / LD
- 大促 BD / LD
- 专享
- 大促基础类型

写入后：

- `type = 'BD（预测）'`
- `coefficient` 使用 `sales_coefficient` 中 `type = 'BD'` 的系数。
- `maybe_sales = weighted_sales * coefficient`

## 6. 合计行 BD 预测优先级

| 优先级 | 类型 | 说明 |
|---|---|---|
| 1 | 真实 BD / LD | 最高优先级 |
| 2 | 大促 BD / LD | 真实活动在大促期间的修正类型 |
| 3 | 专享 | `7月PD(专享)`、`10月秋促(专享)`、`黑五网一(专享)` |
| 4 | BD（预测） | 只补充没有真实活动、没有专享的未来日期 |
| 5 | 基础类型 | 日常、淡季、淡季过渡 |

## 7. 步骤5：修正合计行库存

在合计行 `BD（预测）` 已经更新 `type`、`coefficient`、`maybe_sales` 后，重新递推合计行未来库存。

主要更新：

- `inventory`
- `days_for_sale`
- `sale_inventory`
- `estimate_days_for_sales`

说明：

- 这一步必须放在合计行 BD 预测之后。
- 否则预测 BD 带来的销量变化不会反映到合计库存推演中。

## 8. 步骤6：计算历史加权基准销量

以每条数据自己的 `date` 作为当天，回看过去 30 天到前 1 天，计算历史加权基准销量。

说明：

- 用于补齐历史区间的 `weighted_sales`。
- 与未来预测推演分开处理。

## 9. 步骤7：更新预估销量

最终统一刷新：

- `maybe_sales = weighted_sales * coefficient`

说明：

- 这是收尾步骤。
- 确保日类型和系数变化后，预估销量同步更新。

## 10. 验证清单

| 验证项 | 判断标准 |
|---|---|
| 分店铺基准销量 | 店铺行 `base_sales` 按销量和系数更新 |
| 分店铺加权销量 | 今天及未来店铺行 `weighted_sales` 有值 |
| 分店铺库存 | 店铺行未来库存、可售天数、在途被递推 |
| 合计行生成 | `shop='合计'` 行存在并按国家、ASIN、日期汇总 |
| 合计行日类型 | 合计行按分店铺最大系数确定基础日类型 |
| BD 预测位置 | `BD（预测）` 只出现在合计行 |
| BD 预测间隔 | 欧洲空 7 天，非欧洲空 21 天 |
| BD 预测周期 | 每轮最多 14 天 |
| 阻断规则 | 预测 14 天范围内有真实活动或专享时整轮跳过 |
| BD 预测系数 | `BD（预测）` 使用 `BD` 系数 |
| 合计库存重算 | 合计行预测销量变化后库存重新递推 |
| 最终预估销量 | `maybe_sales = weighted_sales * coefficient` |
