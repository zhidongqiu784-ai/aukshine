# 发货计划演变 v2 当前进度说明

更新时间：2026-07-21

本文档用于新会话接手时快速理解当前「发货计划演变 v2」的实现状态、算法口径、工作流状态和下一步测试方式。

## 1. 当前目标

发货计划演变 v2 的目标是：

把原来销售手填的模拟发货计划，逐步改成系统按新算法自动生成 `shipment_plan_v2` 计划，并用这批计划叠加真实在途数据，推演到 `daily_sales.v2_*` 字段，再由 `发货计划演变.js` 展示表格和图表。

页面上用户拖动节点或改数量后，不是直接改真实数据，而是进入审批流程；审批生效后再写回 `simulate_shipment` 并重算对应 ASIN 的 v2 水位。

## 2. 当前整体链路

目前链路分为四层：

1. 每周生成模拟计划
   - 工作流：`每周生成模拟计划`
   - 作用：生成或更新 `simulate_shipment.plan_source = 'shipment_plan_v2'` 的 W1-W7 模拟发货计划。

2. v2 水位推演
   - SQL：`水位表/工作流：更新水位表/10-发货计划演变-v2水位推演.sql`
   - 作用：把真实在途 + 新算法模拟计划推演到 `daily_sales.v2_*` 字段。

3. JS 页面展示
   - 文件：`水位表/新备货/发货计划演变.js`
   - 作用：展示表格、W1-W7 节点、蓝线/黄线图表、拖动模拟和审批按钮。

4. 审批生效
   - 工作流：
     - `发货计划演变-v2-提交申请`
     - `发货计划演变-v2-审核流转`
     - `发货计划演变-v2-应用计划`
   - 作用：提交修改申请、审核流转、最终写回计划并重算水位。

## 3. 当前算法口径

### 3.1 计划生成对象

当前新算法只生成区域合计计划：

```text
shop = 合计
plan_source = shipment_plan_v2
```

即按「国家 + ASIN + 合计」生成，不再按每个具体店铺生成一份计划。

这是因为实际业务里未来哪一天由哪个店铺卖并不确定，水位判断主要看区域合计更合理。

### 3.2 W1-W7 发货日

W1 从下一周一开始，后续每周加 7 天。

例如当前周期锚点为 2026-07-20，则：

```text
W1 = 2026-07-27
W2 = 2026-08-03
W3 = 2026-08-10
W4 = 2026-08-17
W5 = 2026-08-24
W6 = 2026-08-31
W7 = 2026-09-07
```

### 3.3 预计入库日

预计入库日公式：

```text
预计入库日 = 发货日 + 物流天数 + 入仓天数
```

物流天数来自 `v3_cfg_logistics_lead`。

入仓天数来自 `v3_cfg_cycle_param`，并按淡季/旺季取不同值。

旺季判断区间在 SQL 中按到港/到仓前的日期月份区间判断，例如 9 月中旬会触发旺季入仓天数。

### 3.4 建议数量

计划数量不是简单固定倍数，也不是按店铺均摊。

核心逻辑是：

1. 找到这批货预计入库后真正能服务的销售周。
2. 读取该服务周的系统预测销量：
   - 优先 `daily_sales.maybe_sales`
   - 没有则用 `daily_sales.weighted_sales`
3. 读取服务周起点库存：
   - 使用 `daily_sales.v2_inventory`
4. 如果库存不足，则生成建议数量。

非下单周大致公式：

```text
建议数量 = max(0, 服务周销量需求 × 2 - 服务周起点库存)
```

下单周会额外考虑上一批、下一批的衔接，避免重复下单或漏下单。

### 3.5 真实在途与模拟计划

v2 水位推演读取两类补货：

1. 真实在途
   - 来源口径：`expected_inventory`
   - 条件：`qty_shipped > 0` 且 `remaining > 0`

2. 新算法模拟计划
   - 来源：`simulate_shipment`
   - 条件：`plan_source = 'shipment_plan_v2'`
   - 当前只取 `shop = '合计'`

旧销售手填模拟计划不进入 v2 推演。

```text

| W3 | 2026-08-10 | 海运普线-37天 | 14 | 2026-09-30 | 910 |
| W4 | 2026-08-17 | 海运普线-37天 | 14 | 2026-10-07 | 455 |

```

## 8. 当前未完全验证的功能

尚未用真实业务节点完整端到端测试：

1. 销售拖动节点提交申请。
2. 主管审核。
3. 采购/终审审核。
4. 自动触发应用计划。
5. 写回 `simulate_shipment`。
6. 重算 `daily_sales.v2_*`。
7. 页面刷新后看到新曲线。

原因：

这些动作会创建真实申请记录、写日志，最终可能写回真实新算法计划，因此之前没有擅自用生产数据测试。

## 9. 下一步建议测试方式

建议选择一个可接受产生测试申请的 ASIN，走最小闭环：

1. 打开发货计划页面。
2. 找一个 `shipment_plan_v2` 节点。
3. 小幅拖动数量或日期。
4. 填写理由并提交。
5. 看是否生成 `shipment_plan_change_v2` 申请。
6. 用对应角色审核。
7. 审核到最终应用后，检查：
   - `simulate_shipment` 是否更新。
   - `shipment_plan_change_v2` 状态是否变为 `APPLIED / OK`。
   - `daily_sales.v2_*` 是否重算。
   - 页面刷新后图表是否变化。

如失败，优先查看：

- 工作流执行记录
- 失败节点
- SQL 返回错误
- JS 提交时的提示信息
