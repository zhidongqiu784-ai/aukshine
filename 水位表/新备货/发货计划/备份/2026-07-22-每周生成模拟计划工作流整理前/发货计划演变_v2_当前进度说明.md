# 发货计划演变 v2 当前进度与新会话交接

更新时间：2026-07-22

用途：新开 Codex 会话时，优先让新会话完整读取本文，再读取本文列出的关键文件。本文只记录已经实现、已经核对或明确待验证的事实，不把推测写成完成状态。

## 1. 项目目标

把原来由销售手填的模拟发货计划，逐步替换为系统自动计算的 `shipment_plan_v2` 发货建议，并把真实在途与新算法建议共同推演到 `daily_sales.v2_*`，最后由 NocoBase JS Block 页面展示和支持人工调整审批。

核心原则：

- 计划按 `country + asin + shop=合计` 生成，未来具体由哪个店铺销售不在本模块分配。
- 页面主数字代表“算法建议还应发多少”，不再把真实在途加进主数字。
- 真实在途仍然参与未来库存和安全库存天数推演。
- 人工拖动只修改算法建议，不修改真实在途。
- 旧水位字段和旧销售手填模拟计划保持隔离。

## 3. 各表职责和数据口径

### 3.1 `daily_sales`

用途：提供系统预测销量、真实库存基础和 v2 推演结果。

关键字段：

- `maybe_sales`：系统预计销量，计划需求计算优先使用。
- `weighted_sales`：加权销量，`maybe_sales` 为空时回退使用。
- `inventory`：现有水位表库存字段。
- `days_for_sale`：现有水位表安全库存天数。
- `v2_add`：v2 当天入库量。
- `v2_inventory`：v2 推演库存。
- `v2_days_for_sale`：v2 推演安全库存天数。
- `v2_on_the_way`：v2 推演在途。

页面过去日期优先显示真实 `days_for_sale`，未来日期显示 `v2_days_for_sale`。

### 3.2 `simulate_shipment`

用途：保存系统生成的模拟发货建议。

v2 隔离条件：

```text
plan_source = shipment_plan_v2
shop = 合计
```

主要字段：

- `number`：算法建议发货数量。
- `date`：建议发货日期。
- `add_date`：预计完成入库日期。
- `channel`：物流渠道。
- `warehouse_days`：入仓天数。
- `season`：淡季或旺季。

`number` 是页面格子和图表节点应该显示的主数字。

### 3.3 `expected_inventory`

用途：提供真实在途。

当前有效真实在途条件：

```text
qty_shipped > 0
remaining > 0
```

其中：

- `remaining` 是仍未入库的真实在途数量。
- `expected_storage_time` 是预计入库日期。
- `qty_shipped = 0` 的销售旧模拟计划不进入 v2 真实在途。

### 3.4 `water_product`

继续负责：

- 产品标签 `product_label`
- 预警状态
- 库销比
- 合计未交货 `quantity_receive`

不再作为 v2 图表库存、安全库存天数和真实在途的主要来源。

## 4. 每周计划生成算法

### 4.1 周期锚点

当前系统锚点以 2026-07-06 为下单周，之后按两周节奏判断下单周和非下单周。

每次运行仍以本周一作为 `run_monday`，事实销量和库存读取当日最新快照。

### 4.2 W1-W7

备货当周不进入 W1，W1 从下一周一开始，后续每周增加 7 天。

以 2026-07-20 所在周期为例：

| 周次 | 发货日 |
|---|---|
| W1 | 2026-07-27 |
| W2 | 2026-08-03 |
| W3 | 2026-08-10 |
| W4 | 2026-08-17 |
| W5 | 2026-08-24 |
| W6 | 2026-08-31 |
| W7 | 2026-09-07 |

### 4.3 预计入库日

```text
预计入库日 = 发货日 + 物流天数 + 入仓天数
```

- 物流天数读取物流渠道时效配置。
- 入仓天数读取 `v3_cfg_cycle_param`。
- 淡季使用 `warehouse_off`。
- 旺季使用 `warehouse_peak`。

### 4.4 服务周

服务周不是发货周，而是该批货完成入库后实际承担销量的日期区间。

```text
第一服务周 = add_date 后第 7 天至第 13 天
```

算法要求服务周在 `daily_sales` 中有完整 7 天数据，否则候选会被标记为 `FIRST_SERVICE_WEEK_INCOMPLETE`，不会当作 READY 写入。

### 4.5 需求和起点库存

第一服务周需求：

```sql
SUM(COALESCE(daily_sales.maybe_sales, daily_sales.weighted_sales, 0))
```

服务周起点库存：

```text
服务周第一天的 daily_sales.v2_inventory
```

如果 `v2_inventory` 缺失，算法不会回退到旧 `inventory`，而是标记 `START_INVENTORY_MISSING`。

### 4.6 非下单周 A1

```text
A1 = ceil(max(0, 第一服务周 7 天需求 × 2 - 第一服务周起点 v2 库存))
```

这代表以约 14 天需求为目标，减去货物开始服务时已有的有效库存。

### 4.7 下单周 A2

下单周必须先找到上一周已经生成的 `shipment_plan_v2` A1。

核心口径：

```text
A2 = max(0,
         第一服务周需求 + 第二服务周需求
         - 已扣除上一批承诺后的有效库存
         - 上一批已承诺数量)
```

SQL 还会检查：

- 上一周 A1 是否存在。
- 两个服务周是否都有完整 7 天预测。
- 预计入库日是否晚于最新计划。
- 本运行周是否已经生成。
- 是否存在审批中或已应用的计划变更。

### 4.8 更新插入规则

每周写入 SQL 使用更新插入，不再先清空：

- 同一候选计划已存在且没有活动中的修改申请：更新原记录。
- 候选计划不存在：新增记录。
- 已绑定真实货件号或存在活动修改申请：不覆盖。
- 不读取、不更新旧版计划。

| 周次 | 发货日 | 预计入库日 | 建议数量 |
|---|---|---|---:|
| W1 | 2026-07-27 | 2026-09-09 | 448 |
| W2 | 2026-08-03 | 2026-09-16 | 703 |
| W3 | 2026-08-10 | 2026-09-30 | 910 |

| W6 | 2026-08-31 | 2026-10-21 | 455 |
| W7 | 2026-09-07 | 2026-10-21 | 0 |

这些数据直接来自 `simulate_shipment.number`，不是页面临时计算。

### 5.2 W3 显示口径示例

W3 同时存在：

```text
算法建议：910
真实在途：640，预计 2026-08-11 入库
预计总入库影响：1,550
```

页面当前正确口径应为：

- 表格主数字：`910`
- 图表节点：`W3 · 910`
- 小字或悬浮框：真实在途 `640`
- 悬浮框可展示预计总入库影响 `1,550 = 640 + 910`
- 库存和安全库存天数曲线继续同时考虑 `640 + 910`

因此页面主数修改只是展示口径调整，不是修改生成算法或库存推演公式。

## 9. 正常运行顺序

### 9.1 正常定时链路

1. 现有水位表工作流更新销量、库存和未来基础数据。
2. 工作流节点 10 重算 `daily_sales.v2_*`。
3. 每周生成模拟计划工作流在其计划时间运行。
4. 每周 SQL 更新插入一周新建议或刷新允许更新的建议。
5. 再运行 v2 水位推演，让新计划进入曲线。
6. JS 页面刷新后读取最新建议和 v2 水位。

### 9.2 人工修改链路

1. 用户展开图表并拖动节点。
2. 页面只做沙盘预览。
3. 用户填写理由并提交申请。
4. 对应角色审核。
5. 应用计划工作流写回 `simulate_shipment`。
6. 定向重算目标水位。
7. 页面刷新后显示新建议和新曲线。

```sql
SELECT
    country,
    asin,
    shop,
    number,
    DATE(`date`) AS ship_date,
    DATE(add_date) AS add_date,
    channel,
    warehouse_days,
    plan_source
```
