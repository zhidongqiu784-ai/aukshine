# 发货计划演变 v2 当前进度与新会话交接

更新时间：2026-07-24

当前代码基线：`12c861b 完善发货计划重算与V2水位推演`

用途：新开 Codex 会话时，优先完整读取本文，再读取本文列出的正式文件。本文只记录已经实现、已经核对或明确待验证的事实；动态生产数据均标注核对日期，不把推测写成完成状态。

## 1. 项目目标

把原来由销售手填的模拟发货计划，逐步替换为系统自动计算的 `shipment_plan_v2` 发货建议，并把真实在途与新算法建议共同推演到 `daily_sales.v2_*`，最后由 NocoBase JS Block 页面展示并支持人工调整、审核和应用。

核心原则：

- 计划按 `country + asin + shop = 合计` 生成，不在本模块分配未来具体销售店铺。
- 页面主数字只代表算法建议发货量，不把真实在途加进主数字。
- 真实在途仍参与未来库存和安全库存天数推演。
- 人工拖动只修改算法建议，不修改真实在途。
- 旧销售手填模拟计划与 `shipment_plan_v2` 保持隔离。
- V2 推演与普通库存一致：缺货期间未成交需求不结转，卖完即停止扣减欠货需求。

## 2. 当前正式链路

```text
销售、库存、系数、物流和入仓时效基础数据
        ↓
工作流：更新水位表
节点 01～09 更新普通水位
节点 10 全量重算 daily_sales.v2_*
        ↓
工作流：每周生成模拟计划
节点 01 生成或更新 shipment_plan_v2
节点 02 再次全量重算 daily_sales.v2_*
        ↓
发货计划演变.js 读取计划、真实在途和 V2 水位
        ↓
拖动建议量或日期 → 提交申请 → 审核流转
        ↓
应用计划工作流写回 simulate_shipment
并定向重算对应 country + asin 的 V2 水位
```

算法调整后的已有计划还有一条人工维护链路：

```text
工作流：更新水位表
        ↓
工作流：发货计划演变-v2-重算现有模拟计划
节点 01 重算已有 W1-W7 建议量
节点 02 全量重算 V2 库存与在途
```

## 3. 当前数据表职责

### 3.1 `daily_sales`

提供销量、普通水位基础和 V2 推演结果。

关键字段：

- `maybe_sales`：预计销量，需求计算优先使用。
- `weighted_sales`：加权基准销量，`maybe_sales` 为空时回退使用。
- `inventory`、`days_for_sale`、`add`、`on_the_way`：普通水位口径。
- `v2_add`：V2 当天入库量，包含有效真实补货与新算法模拟计划。
- `v2_inventory`：V2 推演库存。
- `v2_days_for_sale`：V2 推演安全库存天数。
- `v2_on_the_way`：V2 推演在途。
- `v2_calculated_at`：V2 本次推演时间，用于避免较早任务覆盖较晚结果。

页面过去和未来销量表均已有“含 V2 列”版本：

```text
水位表/水位表详情/水位表_销量表格JS版含v2列.js
```

旧版已保留为：

```text
水位表/水位表详情/水位表_销量表格JS版不含v2列.js
```

### 3.2 `simulate_shipment`

保存系统生成的模拟发货建议。

V2 隔离条件：

```text
plan_source = shipment_plan_v2
shop = 合计
```

主要字段：

- `number`：建议发货数量，也是页面格子和图表节点的主数字。
- `date`：建议发货日期。
- `add_date`：预计入库日期。
- `channel`：物流渠道。
- `warehouse_days`：入仓天数。
- `season`：淡季或旺季。
- `v2_calculation_snapshot`：生成或重算时保存的公式输入、目标库存和建议量计算数据。

页面已经移除“（生成快照）”字样，但仍读取 `v2_calculation_snapshot` 展示计算过程。该字段是数据库中的计划计算依据，不是页面临时数据。

### 3.3 真实在途来源

普通水位和页面当前使用 `expected_inventory`：

```text
qty_shipped > 0
remaining > 0
```

- `remaining`：仍未入库的真实在途数量。
- `expected_storage_time`：预计入库日期。
- `qty_shipped = 0`：旧销售手填模拟计划，不进入 V2 真实在途。

但是截至 2026-07-24，三个正式 V2 推演 SQL 的真实在途实现并未直接读取 `expected_inventory`，而是内联使用：

```text
delivery_note.quantity_shipped - fba_ship.received
```

涉及：

- `工作流：更新水位表/10-发货计划演变-v2水位推演.sql`
- `工作流：每周生成模拟计划/02-发货计划演变-v2水位推演.sql`
- `工作流：发货计划演变-v2-应用计划/02-重算对应ASIN水位.sql`

这会造成普通在途、页面真实在途和 V2 在途可能来自不同计算链路。团队经验口径要求 V2 真实在途以 `expected_inventory` 为准；当前尚未统一，属于待处理风险，不能写成已经解决。

### 3.4 `shipment_plan_change_v2` 与日志表

- `shipment_plan_change_v2`：保存修改申请、审核状态、乐观锁信息和应用结果。
- `shipment_plan_change_log_v2`：保存提交、审核和应用过程日志。

## 4. 周次、到货和覆盖售卖期

### 4.1 W1-W7

备货当周不进入 W1，W1 从下一周一开始，后续每周增加 7 天。

以 2026-07-20 所在周为例：

| 周次 | 发货日 |
|---|---|
| W1 | 2026-07-27 |
| W2 | 2026-08-03 |
| W3 | 2026-08-10 |
| W4 | 2026-08-17 |
| W5 | 2026-08-24 |
| W6 | 2026-08-31 |
| W7 | 2026-09-07 |

### 4.2 预计入库日

```text
预计入库日 = 发货日 + 物流天数 + 入仓天数
```

- 物流天数读取物流渠道时效配置。
- 入仓天数读取 `v3_cfg_cycle_param`。
- 淡季使用 `warehouse_off`。
- 旺季使用 `warehouse_peak`。

### 4.3 覆盖售卖期

```text
覆盖售卖期 = add_date 后第 7 天至第 13 天
```

例如 `add_date = 2026-09-09`：

```text
覆盖售卖期 = 2026-09-16 ～ 2026-09-22
```

页面只在周列表头展示覆盖售卖期，不在每个单元格重复显示。详细计算过程放在点击计划后打开的抽屉中。

## 5. 当前建议量算法状态

这里必须区分“每周新生成”和“重算已有计划”，两者截至 2026-07-24 还没有完全统一。

### 5.1 每周新生成：仍是 v3.5 公式

正式文件：

```text
水位表/工作流：每周生成模拟计划/01-生成模拟计划.sql
```

当前公式版本：

```text
shipment_plan_v2_weekly_v3.5
```

非下单周仍按以下方式计算：

```text
建议量 = ceil(max(0, 第一服务周需求 × 2 - 第一服务周起点 v2 库存))
```

下单周仍使用第一、第二服务周需求和上一批承诺数量计算。该节点负责生成新一周计划，但尚未切换到后述“缺货不结转”的重算公式。

### 5.2 重算已有计划：最新缺货不结转公式

正式文件：

```text
水位表/工作流：重算现有模拟计划/01-重算现有W1-W7建议数量.sql
```

当前公式版本：

```text
v2_stockout_no_backlog_2026_07_23
```

核心口径：

```text
覆盖期 7 天需求 = 覆盖售卖期内预计销量合计
目标库存 = 覆盖期 7 天需求 × 2
建议量 = MAX(0, 到货日应有库存 - 到货日前有效库存)
```

其中到货日前有效库存已经包含前面周次重算后的计划补货，但不把缺货期间未成交需求累计成欠货。

### 5.3 当前未统一项

如果只运行“每周生成模拟计划”，新生成周次仍使用 v3.5 公式；如果随后运行“重算现有模拟计划”，已有 W1-W7 会再按最新缺货不结转公式更新。

因此当前状态是：

- 已有计划可以按最新公式重算。
- V2 水位已经使用缺货不结转递推。
- 每周新生成节点尚未完全切换到相同公式。

在统一前，不能把“每周生成模拟计划已经完全采用最新建议算法”写成完成状态。

## 6. V2 库存和在途递推

### 6.1 今日锚点

节点 10 位于普通水位节点 09 之后。普通合计行今日 `inventory` 已包含旧 `add`，因此 V2 先执行：

```text
今日真实库存锚点 = inventory - old add
今日 V2 库存 = 今日真实库存锚点 + 今日 v2_add
```

### 6.2 未来库存

与普通库存缺货口径一致：

```text
如果前一天库存 - 前一天需求 < 0，且今天有正补货：
    今天 V2 库存 = 今天的正补货量
否则：
    今天 V2 库存 = 前一天库存 - 前一天需求 + 今天补货
```

因此库存为 0 后不会继续累计负库存，也不会把缺货期间未成交需求从下一批补货中扣回。

### 6.3 负补货和在途

- `v2_add > 0`：到货，库存增加，在途减少。
- `v2_add < 0`：补货修正，库存减少，在途恢复等量数量。
- `v2_on_the_way` 按“未来正补货总量减去截至当天累计净补货”递推。

### 6.4 安全库存天数

```text
v2_inventory <= 0      → v2_days_for_sale = 0
weighted_sales > 0     → FLOOR(v2_inventory / weighted_sales)
其他情况               → 0
```

## 10. 执行时间和顺序

### 10.1 每日自动链路

```text
08:30  自动校准日类型的系数
08:30  每日更新在途数据
        两个任务同时启动

08:50  更新水位表
        承接系数和普通在途结果
        节点 10 全量重算 V2

之后  每小时 :50 再运行一次更新水位表
```

2026-07-23 的最近记录显示：两个 08:30 任务均在约 08:31 完成，08:50 的水位更新能够承接结果。

### 10.2 每周自动链路

```text
周日 23:50  更新水位表
周一 00:08  每周生成模拟计划
             节点 01 生成建议
             节点 02 重算 V2
周一 00:50  下一次更新水位表
```

当前周任务使用周日 23:50 的水位结果，不会等待周一 00:50。时间差 18 分钟，但不是严格的“周一当天先更新水位再生成计划”。是否调整时间尚未决定。

### 10.3 手工运行：生成新计划

```text
1. 执行“更新水位表”
2. 等待成功
3. 执行“每周生成模拟计划”
4. 检查节点 01 和节点 02 均成功
5. 刷新页面核对建议量和 V2 曲线
```

“更新水位表”只刷新普通水位和 V2 水位，不会重新计算已有建议量。

### 10.4 手工运行：按最新算法重算已有计划

```text
1. 执行“更新水位表”
2. 等待成功
3. 执行“发货计划演变-v2-重算现有模拟计划”
4. 节点 01 重算已有 W1-W7
5. 节点 02 重算 V2 库存与在途
6. 刷新页面核对建议量、库存和在途
```

“每周生成模拟计划”和“重算现有模拟计划”不是固定串行的两个步骤：

- 需要生成新一周计划时运行前者。
- 只需把已有计划切换到最新重算公式时运行后者。
- 未明确目的时不要连续运行两个工作流，否则后运行者会再次更新已有建议。

### 10.5 人工修改和审批

```text
用户拖动节点并提交申请
→ 提交申请工作流校验并创建申请
→ 主管、采购、终审按状态审核
→ 审核流转工作流推进状态
→ 应用计划工作流自动触发
→ 写回 simulate_shipment
→ 定向重算对应 ASIN 的 V2 水位
→ 标记应用完成
```

```sql
SELECT
    country,
    asin,
    shop,
```
