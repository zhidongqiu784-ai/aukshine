## 第 1 章 · 系统在算什么、为什么这么算

### 1.1 业务背景:今天下的单,管的是 3-4 个月后

AOK 主营亚马逊 FBA 自营业务。一台货从下单到能卖,要走完整链条:

```
工厂生产 → 海运到亚马逊仓 → 亚马逊上架扫码 → 真正可以卖
```

这条链最长能拖到约 4 个月(DE:工厂 45 + 海运 60 + 入仓 14 天)。**所以今天下的订单,服务的是 3-4 个月后那两周的销售**,不是当下。这是整套系统所有"提前量"的来源。

### 1.2 老问题:全靠人工估算,两头错

以前用 Excel:靠经验人工估算、缺统一测算口径,填数和审批都没有一致的标尺。结果旺季容易备多 → 滞销压资金;淡季容易备少 → 断货丢排名。

### 1.3 ERP 改了什么:从"填数字"到"复核数字"

ERP 接管"算合理数字"。**系统主导、机制前置**——常规路径系统自动算,销售只做两件事:看 ERP 算的建议、觉得不对就调整(并写理由)。目标是把销售从"填数字"变成"复核数字",省掉大部分逐项审核的负担,同时用统一的测算口径减少人工估算的波动,让备货更稳。

### 1.4 五个必须先懂的概念(只讲含义,公式见字典)

| 概念 | 一句话含义 | 公式 / 口径在字典 |
|---|---|---|
| **备货周期** | 从下单到可售要多久 = 工厂交期 + 运输 + 入仓 + ~~旺季预留 +~~ 安全库存天数<font color="red">(决策 28:旺季异常预留删除;安全库存统一 7/14)</font>。<font color="red">2026-07-07 起用途收窄:三态预警 / 库销比健康区间 / 新品反推仍用;**下单量不再由备货周期锚定的覆盖期公式直接算,改由发货计划倒推净额**(字典 2.5.2 修订)</font> | 字典 2.0.2(参数)、2.3.1(预计可售时间) |
| **库销比** | 当前库存能撑几个月 = **总库存 ÷ 加权基准日均 ÷ 30**(可售月数) | 字典 2.5.2;总库存口径 2.1.5 |
| **库销比健康区间** | ~~从备货周期反推的上下限~~ <font color="red">全局固定 3.5 ~ 4.5(短缺 <3.5 / 滞销 >4.5,决策 28)</font> | 字典 2.5.2<font color="red"> 修订</font> |
| **加权基准日均** | 剔除日类型(BD/旺季)后的"纯日均",滚动当天值 | 字典 2.2.2 |
| **未来每日预估销量** | 已含未来日类型波动的逐日数组,算覆盖期需求用 | 字典 2.2.3 |

> ⚠️ **口径提醒(与旧文档不同)**:库销比分母用「加权基准日均 × 30」**不是**实际累计销量(大促期实际累计会误报短缺);库销比分子用**总库存口径**(字典 2.1.5,含 FBA 可售 + 待调仓 + 海外仓全新 + 检测 + 在途,**不含调仓中 / 入库中**),不是"FBA + 在途"的旧公式。

### 1.5 两个时间维度:淡旺季、新老品

- **淡旺季**(字典 2.0.2):月份粗分=淡季 1-4 月 / 旺季 5-6、9-10 月 / 大促 7-8(PD)、11-12(黑五网一),仅供认知;**系统精确判定以真实活动区间为准**——PD 6/10–7/10、秋促 9/10–10/10、黑五 11/1–12/15(看"抵 FBA 日"是否落进窗口)。旺季**只影响入仓时间**~~和安全库存天数(→ 库销比阈值调高)~~<font color="red">(决策 28:安全库存与库销比阈值均与季节无关)</font>,**不直接放大销量预估**(波动已在每日预估数组里)。
- **新老品**:上架 < 90 天为新品,≥ 90 天为老品。⚠️ 90 天界**不是"两套并行、各跑到底的下单算法"**:开售后所有产品(含 <90 天新品)下单**统一走老品算法**(决策 20,无独立过渡机制)。90 天界只决定三件事——① 上市前首批用新品逻辑(§3.8)② 安全库存天数取新品档(更高,字典安全库存天数表)③ 利润率预警新品期豁免(补 9)。

## 第 2 章 · 角色与职责

### 2.1 三道流程,各管一段(决策 13)

系统把备货从"销售 / 采购 / 老板"改成**销售 / 采购部 / 终审**三道,去人名、去角色绑定:

| 角色 | 审什么 | 职责边界 |
|---|---|---|
| **销售** | 业务合理性(系统建议是否匹配当前推广计划) | 只在异议时触发后续;异议要写理由 |
| **采购部** | ① 质疑销售异议的业务合理性 ② 工厂交期把关 ③ 物流渠道 | 质疑要有据(参考销售历史异议准确率) |
| **终审** | 战略方向 + 异常 | 不看细节,只看汇总指标和标红项 |
| **财务** | **不设独立审核环节**(补 10) | 业务规模未到需严苛财务审核的程度 |

### 2.2 一条铁律:下单严、发货松

这是理解所有流程的总开关——**下单和发货是两件完全不同的事**(决策 14 及其「本期下单 ≠ 发货节奏」补充澄清):

| 维度 | 下单 | 发货 |
|---|---|---|
| <font color="red">推算方向(2026-07-07 反转)</font> | <font color="red">**由发货计划倒推**:净额 = 下单批两周发货合计 − 未交货订单余量(只读结果,字典 2.5.2 修订)</font> | <font color="red">**第一公民**:每周滚动排 1 周(首周承诺 / 次周吸收 / 富余不回改)</font> |
| 决定什么 | 向工厂订多少新货 | 从中国仓发多少到 FBA |
| 频次 | 每 **2 周**一次 | 每 **1 周**一次 |
| 服务的时间 | 约 89 天后那两周(预计可售时间起) | 当下 + 未来几周 |
| 审批 | 销售 → 采购部 → **终审** | 销售 / 系统 → 采购部通知工厂,**不上终审** |
| 可逆性 | 终审通过后**不可取消**(决策 19) | 可调发货时间(W3+ 可调) |

**常见误解**:本期下单的货本周就发了。❌ 错。本期下单的货还在工厂生产,90 天后才到 FBA;**本周从仓库发出去的,是 2-3 个月前下单产生的货**。

> 注:上表"发货不上终审"指**常规发货调整**;**例外照样上终审**——红预警必断货(§3.4)、初期 W3+ 改发货量(§5.1 Q5)。改运输方式(黄预警空派)仍不上终审。

### 2.3 W 周次编号约定(决策 2、6)

发货计划用滚动 W 编号,**关键:W1 = 下周,不是本周**:

| 编号 | 含义 | 谁可调 |
|---|---|---|
| **W0(本周)** | 已发已锁,不进 W 编号体系,只作历史 | 不可调 |
| **W1 / W2(下周 / 下下周)** | 工厂已排产 / 在排料 | ~~销售不可直调~~ <font color="red">守工厂节奏(2026-07-07):销售不可推迟 / 减量,可提前 / 加量 / 改渠道(走主管链);采购可推迟(堆积减压)/ 提前 / 加量 / 改渠道(直改留痕),不可减量(字典 2.4.5 矩阵)</font> |
| **W3 起** | 后续滚动周次 | 两条通道:系统顺延(改时间)+ 销售手动调量<font color="red">(安全区间闸:区间内免审、出界才审,决策 27;字典 2.5.7 修订)</font>;统一口径见 §3.5 |

**为什么 W1 = 下周**:本周的货已发出、改不动,叫 W1 没意义;剥离后销售看到的每个 W 都是"还能改的"。UI 只显示日期范围(如"预计可售时间 8/4-8/17"),不显示 W 编号——销售不需要换算。

## 第 3 章 · 核心业务流程

> 每个流程统一按:**触发 / 频次 → 发起人 → 步骤 → 调用算法(引字典编号)→ 审批 / 留痕 → 为什么**。算法只引编号,真实数字见第 4 章案例。

### 3.1 每 2 周一次:本期下单流程(下单流,需终审)

**触发**:系统每 2 周自动跑一轮;**发起人**:系统算、销售复核。

**步骤**:
1. 系统对每个老品 SKU 算**本期下单量**~~(字典 2.5.2:覆盖期销量 − 滚动到覆盖期起点的库存)~~<font color="red">——2026-07-07 反转:**净额 = 下单批两周发货计划合计 − 未交货订单余量**(发货计划逐周滚动排出,下单周合并;字典 2.5.2 修订)。安全库存已并入每周服务期起点,不再单独加一份。淡旺季 / 入仓 / 活动窗口按**该发货批到货日**判,不再由"今天下单"推算</font>。
2. 系统同时给出**算法解释**(预计可售时间、覆盖期、每日预估销量、库销比、预警等级),让销售一眼看懂"为什么是这个数"。
3. 系统跑**三态预警判定**(字典 2.4.2 / 2.4.3),给 SKU 标绿 / 黄 / 红。
4. 销售复核:
   - 同意 → 一键确认(约 90% 情况)。
   - ~~调整 ≤±15% → 自动通过(补 2)。~~ <font color="red">决策 27(2026-07-07 会议):调整 = 下单异议(改下单批发货格,净额只读跟随),走**安全区间闸**——修改后库存天数落在安全天数区间内 → 免审即时生效(留痕);出界 → 强制理由 + 销售 → 采购部 → 终审,计入异议准确率(字典 2.9.4 修订)。</font>
5. 采购部审(业务合理性 + 交期 + 物流渠道)→ **终审**看汇总 + 标红项 → 一键通过 / 个别调整。
6. 终审通过 = 下单**不可取消**(决策 19),采购部发合同给工厂排产。

**调用算法**:本期下单 2.5.2 · 三态预警 2.4.2/2.4.3 · 可售时间 2.1.3 · 覆盖期销量(每日预估数组 2.2.3)。
**审批 / 留痕**:销售确认 → 采购部 → 终审三道全程留痕(谁、何时、改了什么、理由)。
<font color="red">**切换期影子校验(2026-07-07 补)**:切换后 2~3 个下单周期内,旧下单公式并行只算不用,与净额偏差超 ±20%(可配置)亮灯给采购 / 终审复核、不拦截;连续 3 期干净后下线(字典 2.5.2 修订)。
**下单周时序(2026-07-07 拍板)**:主管批次签核 deadline 在下单周提前至**周二 12:00**(非下单周维持周三 18:00)——PO 周二生成且以签核为前置,原统一 deadline 在下单周形同虚设(字典 2.0.2)。
**承诺偏差指标**:终审看板显示"首周承诺 − 最新重算应发"的当期合计与趋势,旺转淡转折期(10~11 月)重点盯——这是首周承诺机制的代价仪表盘(字典 2.5.2 修订)。</font>
**为什么**:下单不可取消、且服务 3 个月后,错一次代价大,所以走最严的终审;但 90% 是系统算的,销售一键过,不增负担。

### 3.2 每周一次:发货计划监控流程(发货流,不上终审)

**触发**:每周一系统跑发货数量计算;**发起人**:销售在「发货计划演变」页操作。

**步骤**:
1. 系统按发货频次(1 周)算各 W 应发量~~(2 周合计 = 本期下单总量…决策 11 修订)~~<font color="red">——2026-07-07 反转(决策 11 修订退役):**逐周滚动排一周**,非下单周排下单批首周(销售确认即承诺、系统不再重算该周),下单周排次周(次周应发 = max(0, 最新 14 天需求 − 最新起点库存位置 − 首周承诺量),吸收全部漂移);富余不回改,滚入下一循环由下张 PO 自动少下。公式单一源:字典 2.5.2 修订</font>。
2. 销售在发货计划演变页看滚动计划(各 W 的锁定 / 可调 / 可售周高亮等界面呈现见《系统设计》发货计划演变板块,本文不展开)。
3. 销售调 W3+:~~走三道门槛,±15% 内自动过~~ <font color="red">决策 27(2026-07-07 会议):**安全区间闸**——修改后果落在安全库存天数区间内 → 免审自动通过(留痕 + 抄送主管);出界(跌破下限 / 冲破上限)→ 销售 → 主管 → 采购部;下单批出界随 PO 上终审。±15% 与冷却 7 天废除,20% 触发建议保留(字典 2.5.7 修订)</font>。
4. **W1/W2 锁定机制**:系统每周算的 W1/W2 应发量若与原计划有差异——绿 SKU 拦截维持原计划(不调不通知工厂);黄 SKU 走黄预警(§3.3);红 SKU 走红预警(§3.4)。<font color="red">人工通道(2026-07-07 新增):销售可提前 / 加量 / 改渠道(走主管链),不可推迟 / 减量;采购可推迟(堆积减压)/ 提前 / 加量 / 改渠道(直改留痕 + 销售知会),不可减量(字典 2.4.5 矩阵)</font>。

**调用算法**:发货数量计算(决策 11 修订口径)· W3+ 三道门槛 2.5.7 · 预计可售周次映射 2.3.4 · 三态预警 2.4.2 · W1/W2 锁定判定。
**审批 / 留痕**:**W3-W4 若落入本期下单的预计可售周**,其调整会改变下次下单总量,随下单流程审批;否则与 W5+ 同属发货时间调整,不上终审,采购部通知工厂即可。每次调整留痕。
> **W3+ 口径**:本流程的"销售手动调发货量"是 W3+ **两条通道之一(改数量)**,另一条是系统顺延(改时间,§3.5)。统一口径见 **§3.5**;字典 2.3.3/2.5.3/2.5.7 已对齐(2026-06-05),两边一致。
**为什么**:发货不改工厂总产量、只调到货节奏,风险低;W1/W2 工厂已排产/排料,频繁微调成本高,故锁定,只在断货风险时用"改运输方式"而非"改发货量"解决。

### 3.3 黄预警:改运输方式申请流程

**触发**:三态预警判定某 SKU 为**黄**(海运赶不上但空派可救);**发起人**:系统自动算,销售点申请。

**步骤**:
1. 系统在发货计划演变对应 W 格子标 🟡,并自动算**空派量 + 拆分明细**(字典 2.4.8:X 台空派 + Y 台海运)。
2. 销售点"申请改运输方式" → 默认接受系统算的台数(可编辑)→ 填理由(可选)→ 提交。
3. 进采购部审核(「采购部审核」板块"改运输方式申请"分类)→ 通过 / 拒绝。
4. 通过 → 通知工厂改部分批次为空派;拒绝 → 维持原计划(销售可重新申请)。

**调用算法**:三态预警 2.4.2/2.4.3 · 空派数量 2.4.8。
**审批 / 留痕**:采购部审,**不上终审**(改运输方式不改发货总量、不影响订单,与"发货不上终审"一致);留痕。
**为什么**:黄预警是"小事",系统自动算空派量、不增加销售工作量;空运费属运营调整范畴,采购部可决策。

### 3.4 红预警:必断货特殊处理流程

**触发**:三态预警判定某 SKU 为**红**(空派也来不及救);**发起人**:系统判定 + 销售填客观信息。

**步骤**:
1. 系统判红 → 进入「必断货特殊处理」独立流程(决策 9;水位表标红、侧边栏板块等呈现见《系统设计》)。
2. 销售在板块内填**客观信息**:工厂可分批交货明细(批次 / 出厂日 / 数量,跟工厂问)、空海运每台成本差(可选,跟货代问)。
3. 系统算**最优解**(字典 2.4.9:从批次 1 累加、找刚好够撑到海运到货的最小空派组合,优化目标=尽量少发空派);输出推荐方案(空派组合 + 海运量 + 断货天数 + 损失台数 + 额外空运成本)。
4. 特殊情形:工厂无加急(销售不填批次)→ 系统直接出"全程海运"方案;全空派仍不够 → 输出"接受断货"方案,标明断货天数。
5. 销售确认 → **终审** → 执行。
6. 若实际断货 > 7 天,接上后自动启动 **28 天恢复期**(决策 7:前 14 天 ×0.7 / 后 14 天 ×0.85)。

**调用算法**:三态预警 2.4.2/2.4.3 · 空派最优解 2.4.9 · 28 天恢复期 2.4.6(即旧称"函数 K")。
**审批 / 留痕**:**上终审**(涉大额空运费 + 发货策略调整);销售只填客观信息、不做计算决策;系统不算总成本/总损失(毛利由销售自评估);全程留痕。
**为什么**:必断货是高风险场景,独立板块让终审专门关注,不混在常规流程里。

### 3.5 W3+ 顺延流程(销量回落)

**触发**:系统每周一跑 W3+ 检查;**发起人**:系统自动,采购部审。

**步骤**:
1. 对每个**绿色** SKU 模拟:顺延 1 周后库销比是否更接近健康区间?是 → 触发顺延,否 → 维持。
2. 默认动作:整批顺延 1 周 → 内部自检(滚动模拟顺延后会不会断货):不会 → 整批顺延;会 → 部分保留(够撑到下批到货)+ 剩余顺延。
3. 推采购部审核(无需终审)→ 通知工厂。
4. **累计顺延 > 4 周** → 系统提醒"该 SKU 已累计顺延 N 周,要不要重估销量预测?"(提醒,非强制)。

**调用算法**:库销比 2.5.2 · 库存滚动模拟(同预计断货日同源)。
**审批 / 留痕**:采购部审,无终审;留痕。
**为什么**:处理"销量比预期低 / 库存堆积",自动减压,避免反复加码导致库存堆积。**黄 / 红 SKU 不进顺延**(它们要的是补货 / 救急,不是延后),只有绿色 SKU 走。

> **W3+ 统一口径(全文 W3+ 以此为准;字典 2.3.3/2.5.3/2.5.7 已对齐,见 §4.9 发现 ⑦)**:W3+ 有**两条并存通道**——
> ① **系统顺延**(本流程):自动、每周一跑、只对绿色 SKU 库存堆积、改发货**时间**(推迟)、采购部审无终审;
> ② **销售手动调发货量**(§3.2):销售主动、走三道门槛(2.5.7)、改**数量**、初期统一上终审。
> **销售手动调量后,系统必须重跑顺延模拟**(否则顺延依据的库销比已失效)。
> 字典 2.3.3 / 2.5.3 原"W3+ 不由销售直接调"措辞过宽,已于 2026-06-05 收窄为"**时间顺延归系统**"、2.5.7 补"销售调量后重跑顺延"联动——**本口径与字典一致**(时间归系统、数量归销售)。

### 3.6 店铺拆分调整流程

**触发**:本期下单 / 发货需落到具体店铺;**发起人**:系统给参考、人工分配。

**当前范围(决策 23 二次简化,先讲这条)**:系统只在**区域(合计)层**算下单 / 三态预警;**店铺层不做断货精算 / 按需分摊**。店铺层只保留一件事——**主力店("大号 1/2")15 天库存活动保障预警**(FBA 可售 ÷ 日销 < 15 天即亮,保证活动有货)。非主力店断货靠"切同款有货店卖"人工解决;海外仓调拨到店的分配交人工(系统给池余量 + 主力店预警作参考)。

**步骤**:
1. 系统在区域层算出下单 / 发货总量。
2. 落到店铺:当前**交人工分配**,系统提供主力店 15 天预警 + 海外仓池余量作参考。
3. 主力店亮 15 天预警 → 优先保障(为活动备货)。

**调用算法**:库销比 / 可售月数 2.5.2 · 主力店保障预警(决策 23 口径)。
**审批 / 留痕**:随发货流走;留痕。
**为什么**:全店总库存正常但单店可能一边断货一边积压,本想做店铺级精算;但"同款可切店卖 + 突发因素多、精算性价比低",两次简化后定型为"区域层算 + 主力店保障预警"。⚠️ **已作废**:旧"系统按历史比例自动拆分 + 销售详情页逐店调 + B 店占比"那套店铺级分摊流程(架构 6.6)已被本决策取代,不要回用。完整店铺维度后置、按实际需要再补。

### 3.7 库存健康看板:滞销 / 淘汰流转流程

**触发**:库销比 > 动态上限(滞销)持续 + 利润不达标;**发起人**:系统判定推采购部。

**步骤**:
1. 系统按库销比判**滞销态**(字典 2.5.6:库销比 > 动态上限;**旧"风险四级 + 售出率"模型已撤回作废**,补 11);已判滞销的 SKU 内,用库龄分段(91-180/181-270/271-365/365+)做店铺定位 + 清货优先级排序(展示逻辑,不参与判定)。
2. **淘汰预警**(决策 21):**滞销 + 利润双条件(AND)**——① 库销比 > 上限(滞销)② 连续 2 月利润率 < 10%;**且排除** ③ 断货 / 恢复期态(这两态月销被压低、库销比虚高,排除以免把断货误判成滞销)→ 才推采购部。
3. 采购部跟销售 + 产品双向确认 → 拍板:确认淘汰 → 产品标签打"淘汰期"→ 停备货、移出备货池 →(交清货体系);继续观察 → 一个月一次,再次亮预警 → 升级会议定夺。
4. 观察期内库销比回健康区间 → 清空观察态、解除预警。

**调用算法**:库销比 2.5.2 · 滞销判定 2.5.6 · 淘汰双条件 2.5.4。
**审批 / 留痕**:采购部居中拍板(不让被追责的销售自己拍);"确认淘汰 / 继续观察 / 会议决定"留痕,供与清货体系对账。
**为什么**:滞销 ≠ 淘汰。只滞销可能只是短期备多(健康);双条件 + 人工确认 + 会议升级,既不误杀健康品、也不让单一指标自动清货。淘汰期后的清货执行归清货体系,不在本系统。

### 3.8 新品首批备货流程(下单 3 次)

**触发**:新品上市前;**发起人**:销售填 4 个客观字段,系统算。

**步骤**:
1. 销售填:项目方出货日期、目标站点、对标老品(必填,系统据此抓月度销量历史)、产品等级(从主数据拽,S/A/B,无 C)。
2. 系统算**总量**(字典 2.11;系数见 2.0.2)。总下单量 = 总发货量,无独立应急储备。
3. 系统**分 3 批发货**(按对标历史的递减节奏,占比口径见 2.11)→ 反推 **3 次工厂下单日**(每次约间隔 30 天)。
4. 走销售 → 采购部 → 终审。
5. **下单 3 次的天然缓冲**:任意时刻发现项目降级 → 不启动后续下单 → 自动减压货风险。
6. 开售后:**开售当天即并入老品算法**(决策 20,无独立过渡机制),实际销量从第 1 天起喂下单 / 发货 / 三态 / 顺延;前两批库存缓冲替代"等数据攒稳"。

**调用算法**:新品首批总量 2.11.4 · 发货分批 2.11.5 · 下单 3 次 2.11.6 · 系数 2.0.2 / 2.11.3。
**审批 / 留痕**:三道全留痕。
**为什么**:新品无历史,逻辑跟老品相反——先一次性下足总量、用 3 批发货吸收销量不确定性;销售推新系数**只减不加**(系统基线已偏激进,不再给加码通道)。
> 注:新品首批用的"产品等级系数(S/A/B)"是 SABC 等级的**新品首批专用例外**——**老品下单不用 SABC**(已退出算法,2.8),只有新品首批用它(2.11.3)。两处别混(FAQ Q6 同此)。

### 3.9 跨站未交货订单改派流程

**触发**:A 站本期有下单缺口 且 B 站同型号有滞销 / 富余未交货订单;**发起人**:系统出建议,采购部判。

**前提门槛(决策 24)**:① 海外仓成品库存**跨区域不互调**(地理远);② 只有**国内工厂端未交货订单**可改目的地。改派只动还没交货的订单,不碰已入仓成品。

**步骤**:
1. 系统出「跨站改派建议」:B 站未交货订单改目的地到 A,A 站本期下单相应减少(建议改派量口径见决策 24;B 站可释放余量须排除本次待判断的未交货订单,防 B 站被判"库存够"改走后反断货;**B 站处于断货中 / 恢复期不得作为让单方**,字典 2.7.6)。
2. **采购部规格可行性放行(2026-06-10 口径,唯一一道人工触点)**:判规格 / 包装 / 说明书可行(产线已进定制工序不可改),可行即过——改派**默认执行、无审批流**,到货时效不影响改派决定;放行时顺带人工核对 B 站主力店 15 天库存预警(兜底,字典 2.7.6)。
3. 改派执行后**原子化双边重算**:A 加"改派在途"、B 未交货减,两边覆盖期 / 三态 / 下单量同步刷;改派在途落区域层、不落具体店铺。
4. **执行后自动抄送两站销售 + 两站主管**(知会、非确认,不设回执卡点;含改派量 / 对 B 站水位影响 / 异议入口,字典 2.7.6)。

**调用算法**:本期下单 2.5.2 · 三态预警 2.4.2(按改派后路径重算 A 站)· 动作仲裁 2.5.8。
**审批 / 留痕**:无审批流,仅采购部规格放行;放行 + 双边重算 + 抄送全程留痕。
**为什么**:海外仓跨区地理远不互调,跨站协同改走"工厂端改未交货订单目的地"——只动没交货的订单,不碰已入仓成品、不触不可取消红线(改派 ≠ 取消)。**改派不是救急**:A 站红预警必断时改派来的货到不了,救急仍归空派 / 急单。改派是公司库存周转率最优解(B 站未交货订单 = 已占用资金,A 站有真需求即优先消化),故到货时效不作判断条件、审批降为一道规格放行(2026-06-10 决策 24 批注)。

### 3.10 海外仓自动调拨指令流程

**触发**:每周一系统跑(跟发货同频);**发起人**:系统判,采购部审。

**步骤**:
1. 对每个 SKU 滚动模拟未来[窗口]天 FBA 可售(起始 = 当前 FBA 可售 + 国内在途到货 − 窗口内预估销量;**不算海外仓默认在途**——主动决策视角)。
2. 窗口内 FBA 可售有为 0 的天 → 触发调拨;系统算**建议调拨量**(口径见决策 16,约束 ≤ 当前海外仓库存)。
3. 多 SKU 按 FBA 可售见底时间早晚排序;推采购部审核。

**调用算法**:库存滚动模拟 · 调拨窗口(淡季 14 / 旺季 21 天)。
**审批 / 留痕**:采购部审,**无终审**;留痕。
**为什么**:海外仓库存 = 默认在途(决策 15),迟早调到 FBA;调拨判定故意**不算**海外仓默认在途(否则永远不触发),与三态预警(算入,客观会到)是不同视角的双口径。

### 3.11 链接状态联动(异常自动冻结)

**触发**:监听亚马逊后台 / 领星的 Listing 状态变化;**发起人**:系统自动。

**步骤**:
1. 下架 / 暂停 / 审核中 → 立即冻结该 ASIN 全部未发货计划 + 库存标"冻结中"不计入可用 + 即时通知销售采购部 + 进"状态异常清单"每日复核。
2. 恢复在售 → 解冻 + 重新评估(需销售确认)。
3. 被跟卖 / 丢 Buy Box → 预警 + 观察阈值收紧(偏离 >10% 即触发),不自动改计划。

**调用算法**:**字典 2.4.10(链接异常冻结,2026-06-10 已补编号,口径单一源)**——优先级高于 W1/W2 锁定(链接下架时已排产订单也要冻结);冻结期间动作层全停(唯改派让单放行),见字典 2.5.8 仲裁矩阵。
**为什么**:链接异常时系统还按正常销量预估会积压或误判断货,机制兜住"产品没法卖了但系统不知道"的盲区。

### 3.12 数据测算 Excel 离线流程

**触发**:需离线核算 / 给工厂或外部看;**发起人**:采购部 / 销售。

**步骤**:导出当前 SKU 的关键字段 + 系统建议 → Excel 离线查看 / 调整 → 不回写系统(离线工具仅供测算参考,正式调整仍走系统流程)。

**为什么**:保留一个轻量离线核对口,但不让离线表成为第二真相源——所有正式决策仍以系统为准。

### 3.13 异常销量处置流程(决策 22)

**触发**:① 销量突增(系统检测)或 ② 销售发现蹭 BD;**发起人**:系统检测 / 销售反馈,采购部研判。

**步骤**:
1. **销量突增 →「其他异常预警」**:系统按字典 2.5.5 检测(连续若干天实际销量显著高于加权基准日均且达门槛)→ 亮"其他异常预警"(与三态预警、淘汰预警并列,**只提示关注、不直接驱动备货**)→ 推采购部。
2. 采购部研判:① 未识别活动(如绑定变体蹭 BD 量)→ 走下条人工补标;② 真实上涨 → 关注备货,交三态预警 / 覆盖缺口加单(§3.1)。
3. **蹭 BD 人工补标**:销售反馈 → 采购部按实际活动类型给那几天**手动补打日类型** → 系统用日类型系数剥离(机制见字典 2.2)→ 补标留痕。

**调用算法**:异常销量识别 2.5.5 · 加权基准日均 2.2.2 · 日类型剥离(字典 2.2)。
**审批 / 留痕**:采购部研判,手动补标留痕(只影响后续下单,已下单不可取消,决策 19)。
**为什么**:异常销量(尤其无活动标记的突发高销量)会污染加权基准(→ 备货虚高)和库销比(→ 掩盖淘汰预警)。不为少见的蹭 BD 造重型自动识别,复用日类型系数 + 人工补标;用一个轻量"其他异常预警"兜住所有突增,都引向采购部研判。

### 案例 1 · 正常老品下单 · 绿预警(US 海运淡季)

**验**:本期下单 2.5.2 · 三态预警 2.4.2/2.4.3 · 库销比 2.5.2 · 总库存 2.1.5

**设定**:加权基准日均 40(淡季平稳,每日预估≈40);FBA 可售 2600、在途 1500、海外仓 0、未交付 0。

**算**:
- 预计可售时间(2.3.1)= 5/7 + 45 + 37 + 7 = **8/4**;覆盖期起点(2.5.2)= 可售时间 + 安全库存下限 7 = **8/11**;覆盖期 = 8/11–8/24(14 天)。
- 覆盖期销量 = 14 × 40 = **560**。
- 覆盖期起点库存位置(ERP 滚动模拟,2.5.2):(2600 + 1500)− 96 天 × 40 = 4100 − 3840 = **260**。
- **本期下单 = 560 − 260 = 300 台**。
- 三态(2.4.2):预计断货日(FBA 可售口径)= (2600 + 1500) ÷ 40 ≈ 第 102.5 天 ≈ **8/18**;海运到货可售日 = 8/4。8/18 ≥ 8/4 → **🟢 绿**。
- 库销比(2.5.2):总库存(2.1.5)= 在途 1500 + FBA 可售 2600 = 4100;月销 = 加权基准 40 × 30 = 1200;库销比 = **3.42**。

**结论**:公式跑通,绿、下单 300 台。使用者能说清"覆盖期要 560、滚动到起点还剩 260、所以补 300"。
<font color="red">⛔ 决策 28(2026-07-08)后本条口径作废:库销比健康区间不再按站点 × 新老品 × 季派生,全局固定 3.5~4.5(字典 2.5.2 修订),下述数字仅作历史对照。</font>**🔎 发现①(已解决 2026-06-05,~~已被决策 28 覆盖~~)**:US 老品淡季权威区间 = ~~**3.20 ~ 3.43**~~(字典 2.5.2「库销比健康区间表」,45 天交期:下限 (45+37+7+7)/30、上限 (45+37+7+14)/30);库销比 3.42 落健康上限内(边缘)。02 架构 6.9.1 的 3.37~3.60 是 **50 天旧表、已作废**(交期统一 45,决策 1)。口径锁定:健康区间 = 完整备货周期(交期 45+运输+当前季入仓+旺季异常+新老安全)÷30,**按当前季取列**。
> 附:**库销比维度(中长期健康)与三态绿(短期断货)是两个独立维度,可不一致**。本例三态绿(短期不断货)、库销比却在上限边缘(中长期偏多),两者不矛盾——业务逻辑与界面别把它们混为"一个健康灯"。

### 案例 2 · 老品黄预警 · 改空派(US 海运淡季)

**验**:三态 2.4.2 · 空派数量 2.4.8

**设定**:加权基准 40;FBA 可售 2760、无及时在途。

**算**:
- 预计断货日 = 2760 ÷ 40 = 69 天 = **7/15**;空派到货可售日 = 5/7 + 45 + 7 + 7 = 第 59 天 = **7/5**;海运到货可售日 = **8/4**。
- 判定(2.4.2):7/5 ≤ 7/15 < 8/4 → **🟡 黄**。
- 空派数量(2.4.8):缺口期 = 7/15 → 8/4(20 天);缺口期销量 = 20 × 40 = 800;空派缓冲 = 40 × 7 = 280;**空派数量 = 1080 台**。

**结论**:公式跑通,黄、需空派约 1080 台填 7/15–8/4 的窗口。
**🔎 发现②**:空派数量按**缺口期(20 天,跨约 3 个发货周)**算 = 1080;但 2.4.8 的"海运/空派拆分 = max(0, **该周原计划量** − 空派数量)"是按**单周**算,单周发货量才约 280。空派量(跨多周)与拆分(按单周)口径不闭合——**1080 台空派该如何分摊到多个发货周、对应几批空派**,字典没说清。→ 需补"空派量跨多周时的批次分摊口径"。

### 案例 3 · 老品红预警 · 必断货 + 恢复期(US 海运)

**验**:三态 2.4.2 · 空派最优解 2.4.9 · 28 天恢复期 2.4.6

**设定**:加权基准 40;FBA 可售仅 400、无及时在途;工厂可分批:批 1(5/20 出 500)、批 2(6/5 出 500)。

**算**:
- 预计断货日 = 400 ÷ 40 = 10 天 = **5/17**;空派到货 = 5/7 + 45 + 7 + 7 = **7/5**。5/17 < 7/5 → **🔴 红**。
- 空派最优解(2.4.9)需跑**全程库存曲线**(两批都走空派以压缩断货):
  - 批 1 空派 = 5/20 + 7 + 7 = **6/3 到**;断货日 5/17 → 6/3 = **第 1 段断货 17 天**(空派也补不上,这正是"红")。
  - 批 1 的 500 台到 6/3 后撑 500 ÷ 40 ≈ 12.5 天 → 约 6/16 再次见底;批 2 空派 = 6/5 + 7 + 7 = **6/19 到**;6/16 → 6/19 = **第 2 段断货约 3 天**。
  - 全程断货合计 ≈ **20 天**,损失台数 ≈ 20 × 40 = **800**(不是只算第 1 段的 680);额外空运成本 = 两批空派量 × 每台成本差。
- 断货 20 天 > 7 天 → 接上后启动 **28 天恢复期(2.4.6)**:前 14 天每日预估 ×0.7、后 14 天 ×0.85。

**结论**:公式跑通,落"最小化断货损失 + 恢复期"。**这里我第一版只算了第 1 段(680)是错的——红预警损失必须把全程曲线跑完(批 1 到货后再次见底、批 2 才到),才是真损失(≈800)。**
**🔎 发现③**:2.4.9 把优化目标写成"找让全程库存**不破 0** 的最小空派组合",但红预警的前提就是"空派也来不及、必断"——红预警案例几乎必然**没有**不破 0 的组合。所以 J 的真实目标是"**最小化断货天数 / 损失台数**"而非"消除断货",字典表述与红预警前提自相矛盾,需改口径表述。

### 案例 4 · 新品首批 · 下单 3 次(US)

**验**:新品首批总量 2.11.4 · 发货分批 2.11.5 · 下单分批 2.11.6

**设定**:对标老品三月销量 450 / 600 / 750;销售高级(系数 1.0);S 级(系数 1.2)。

**算**:
- 总量(2.11.4)= (450 + 600 + 750) × 1.0 × 1.2 = 1800 × 1.2 = **2160 台**。
- 发货 3 批(2.11.5):发货 1 = 2160 × 750/1800 = **900**(开售前到);发货 2 = 2160 × 600/1800 = **720**(开售 +30);发货 3 = 2160 × 450/1800 = **540**(开售 +60)。
  > 注:字典称"3-2-1 递减"是**俗称**,实际**按对标第 3/2/1 月销量占比**算(本例占比 0.417/0.333/0.25,并非严格 3:2:1)。占比口径以 2.11.5 为准,"3-2-1"只是形象说法。
- 下单 3 次(2.11.6):工厂下单日 = 发货到 FBA 日 − 入仓 − **快船时效** − 产品交期。

**结论**:总量 / 发货分批跑通(2160 / 900-720-540)。
**🔎 发现④**:下单日反推要用**快船时效**(新品默认走快船,2.11.7),但**配置参数表 2.0.2 没有"快船时效"的站点数值**(只有海运 / 空运)。→ 新品 3 次下单日无法精确反推,**缺"快船时效"参数**,需补进 2.0.2 站点时效表。

### 案例 7 · W3+ 顺延 · 绿色 SKU 销量回落(US 老品)

**验**:W3+ 顺延 2.5.3

### 案例 8 · 跨站改派 + 主力店保障(US 缺口 + JP 富余)

### 4.10 边界场景补充(验证算法在临界值不跑偏)

上面 8 个走主干路径。以下 3 个**临界场景**最易出错,单独验一遍:

- **边界 A · 覆盖期起点库存为负 → 放大下单**:同案例 1,但起点库存位置滚动出 **−400**(8/11 那天已缺 400)。本期下单 = 覆盖期销量 560 − (−400) = **960**。验证:公式对负库存正确**放大**下单(补缺口 + 覆盖期),不是只下 560。
- **边界 B · 下单 ≤ 0 → 不下单**:某 SKU 起点库存位置 = **800**,覆盖期销量 = 560。本期下单 = 560 − 800 = **−240 ≤ 0 → 不下单**(2.5.2 三情况之一)。验证:库存厚 / 滞销品自然落"不下单",系统应留原因(库存足),不报错、不下负单。
- **边界 C · 恢复期折减反向影响下单**:断货接上的 SKU,恢复期内每日预估 ×0.7/×0.85(2.4.6),覆盖期销量随之**变小** → 本期下单相应**减少**。验证:恢复期是改"每日预估销量数组"本身,下单 / 发货 / 三态 / 调拨**自动跟随打折值**(2.4.7),不需为下单单独写规则。

**其余边界留作 IT 实现的回归测试用例(对接时必过,对应字典 2.6 / 测试用例精神)**:① 未交付订单只进覆盖期、不进断货日(易混);② 新品非 S 级 + 初级销售(0.8×0.8 叠乘);③ 异常突增但日均 <10 不触发(防低销量误报);④ 空派最优解多批次大小不一时是否真取最小组合(非贪心取最早);⑤ 跨站改派后目标站点仍缺口的双边重算。

## 第 5 章 · 常见问题 + 业务↔算法翻译表

> 面向所有用到备货流程的同事(销售 / 采购 / 跨部门 / 新人)。口径已按 v4 字典校准,具体公式引编号、不在此展开。

### 5.1 FAQ

**Q1 为什么淡季下单量比我感觉的少?**
淡季入仓时间短~~、安全库存天数少、不需要旺季异常预留~~ → 备货周期短 → 需要的库存量就少(参数见 2.0.2)。<font color="red">决策 28 后:安全库存与季节无关(统一 7/14),旺季异常预留已删;淡旺季差异只剩**入仓时间**一项(保留)。</font>

**Q2 为什么旺季下单这么大?**
旺季完整备货周期变长(入仓 + 安全库存天数都变长),要为更长的"等待期"备货;若还要应对 BD,数量更大。

**Q3 我能改 ERP 算的数吗?**
能,规则一句话(决策 27):<font color="red">**改完的后果在安全库存天数区间内 → 系统默认通过,不用等任何人**(留痕 + 抄送主管);**出界才审**——W3+ 发货出界走 主管 + 采购部,下单批出界走 采购部 + 终审(强制理由 + 计入异议准确率)。**W1/W2** 例外,见守工厂节奏矩阵(2.4.5,审的是货源可执行性)</font>;**空派量**可编辑但走采购部审(2.4.8);**新品首批**但凡偏离系统测算量就要写理由、走新品流程(2.11.9)。~~别用一个 ±15% 套所有。~~<font color="red">±15% 已整体退役。</font>

**Q4 订单下出去能取消吗?**
不能(决策 19)。终审通过 = 不可取消,所以下单才走最严的终审。实在不需要这么多,只能"调整发货计划"延迟发货,不能退单。

**Q5 发货调整要终审吗?**
分两种:**改运输方式**(黄预警空派)不上终审,采购部审即可(2.4.3);**W3+ 改发货量**初期统一上终审(2.5.7,从严管控,后期可配置放开)。
> ⚠️ W3+ 到底谁能调、与系统顺延怎么衔接,字典 2.3.3/2.5.3 与 2.5.7 措辞待对齐(见 §4.9 发现 ⑦),以字典回修后为准。

**Q6 SABC 等级会影响下单量吗?**
**老品不会**。SABC 已退出老品下单算法(2.8),"未来每日预估销量"本身已反映产品销售表现(S 级销量本就高)。SABC 只触发管理动作:C 级停备货、启动清货。
**新品首批例外**:新品首批用"产品等级系数(S/A/B)"算总量(2.11.3 / 案例 4)——那是开售前的事;开售后并入老品,同样不看 SABC。

**Q7 销售职级会影响下单量吗?**
老品**不会**(销售实力已在历史销量里,再乘会重复打折)。只有**新品首批**用销售推新系数(2.11.3),因为新品没历史。

**Q8 库销比 3.5 是好还是坏?**
~~取决于站点 × 淡旺季 × 新老品(健康区间 = 备货周期 ÷ 30)~~ <font color="red">决策 28 后一句话记住:**短缺 <3.5 / 正常 3.5~4.5 / 滞销 >4.5**,全站全品全季一把尺。3.5 恰好踩在短缺 / 正常的分界上——刚好及格。系统标签(短缺 / 正常 / 滞销)按此判,人也可以直接心算了。</font>
> 注:~~字典 2.5.2「库销比健康区间表」~~ <font color="red">派生区间表已作废(决策 28),固定阈值见字典 2.5.2 公式块 / 2.0.2 参数表。</font>

**Q9 为什么有"加权基准日均"和"未来每日预估销量"两个?**
用途不同:**加权基准日均**(剥离活动、稳定基准)用来算安全库存;**未来每日预估销量数组**(含活动波动)用来算覆盖期需求(2.2.3)。

<font color="red">**Q10-A(2026-07-07 新增)"我想晚点发" 和 "我想少订货" 是两回事,别走错格子**
两个意图、同一张表,动作完全不同:**只想调节奏**(货还是要的,晚点到)→ 改发货周的**时间**(W3+ 系统顺延 2.5.3 / W1-W2 找采购推迟 2.4.5),下单净额不受影响;**想少订货**(需求真降了)→ 改**下单批(W6/W7)发货格的数量**(下单异议 2.9.4),净额自动减少。走错的典型事故:想晚点发却去砍了 W6 数量 = 砍了订单;想少订货却只顺延 = 订单原样下出去。净额格的原地推导展示会显示每次改动对 PO 的影响,提交前看一眼。上线培训必讲条目。</font>

**Q10 海外仓库存算我的可用库存吗?**
算"总库存"(库销比用,2.1.5),但**断货预测 / 三态预警用 FBA 可售、不含它**(双口径,2.1.4)。它是"默认在途",7 天调拨 + 入仓后才到 FBA 可售(2.1.3)。
> 注:"在途"含两类——**国内发货在途**(海运/空运中)和**海外仓默认在途**(海外仓库存),都计入总库存但来源不同;海外仓那部分要经 7 天调拨 + 入仓才转成 FBA 可售,别和国内在途混为一谈。
