① 节奏与铁律

一句话：销售每周排一周、确认一周；采购每两周合并两周、下一张 PO。

  • 非下单周：系统排出批次首周（如 W6）应发量 → 销售周一确认 → 即承诺。
  • 下单周：系统用最新数据排次周（W7），滚动模拟把首周承诺量当既定供给滚入；周二采购合并两周下 PO。
  • 每周一前沿链式延伸一格，天然不重不漏。基准示例：7/6 备货周，备货当周不进表，W1 = 7/13 起算；+45 天交期 → 本期下单批 = W6 + W7。

三条铁律（v8 修订版）

铁律内容v8 修订说明
① 首周承诺系统不重算非下单周确认后，系统每周一重算不改首周的数（防横跳、保节奏）。[待拍板] 修订点：承诺 = 系统不重算；人可在周二 PO 落地前提「下单异议」（申请 → 采购 → 终审），异议通过后系统重跑一次 W7 吸收再出净额。真锁点 = PO 落地（决策 19 不可取消）。系统重算若发现首周偏差超阈值，标 🟡 例外格亮灯提示、不改数。
② 次周吸收全部漂移次周应发 = max(0, 最新 14 天需求 − 最新起点库存位置 − 首周承诺量)。不变。它保证 PO 总量 = max(首周承诺, 最新 14 天净额)：需求走高时 PO 完全正确；走低时误差有上限（一周量）且 2~3 循环自愈（定稿附录 A）。
③ 富余不回改销量下修到「次周砍成 0 仍富余」也不回头动首周；富余滚入下一循环，由下一张 PO 自动少下消化。不变。救急仅限黄 / 红预警通道。
PO = 首周承诺 A₁ + max(0, N₂ − A₁) = max(A₁, N₂)　　其中 N₂ = 14·D₂ − S₂（最新 14 天净额，不 floor —— B1 修复前提）

② 表结构导读

表头三行

周次（W1 = 下周，本周已发已锁不进编号）／发货日期（出货 / 出厂轴）／覆盖售卖期（这批发出去补的是哪段可售 ≈ 出货 + 时效 + 入仓）。本期下单批 W6–W7 紫框高亮。

商品信息 9 列

三态点 + 站点（断货标）｜销售｜型号｜ASIN｜加权日均｜等级（S/A/B）｜库销比（短缺 <3.5 / 正常 3.5–4.5 / 滞销 >4.5，决策 28 全局固定）｜生命周期标签（新品 / 成长 / 成熟 / 淘汰）——看一眼初判「要不要备、能不能备」。

周段格子

  • W1–W2：默认折叠成一窄列（如 1000+800 🔒），点开看明细。锁定语义见 ③。
  • W3–W7：正常格 = 裸数字（零装饰）；动过的格 = 标记（↑↓✈⟳）+ 五色流转外框。信息三级递进：格上标记 → 悬停一句话（理由 + 卡在谁 + 截止）→ 抽屉完整时间线。
  • 点任何可改格 = 打开模拟演算（入口归一，见 ⑨）；点例外格 / 待审格 = 打开抽屉。

净额列（只读结果 · PO 唯一依据）

需下单（净额）= W6 应发 + W7 应发 − 未交货订单余量（余量 = 未交货总量 − 已被 W1~W5 计划占用，系统滚动自动扣）

净额是结果不是过程（v10）：销售改 W6 / W7 格子 = 改下单计划本身，净额自动跟着变，因此净额列无修改入口、无确认动作。推导原地展示在净额格内（合计 − 余量 = 需下单 + PO 链状态），格子不可点开；余量口径悬停可见。净额 = 0 → 本批不下新单，发货用既有订单余量。周二采购点顶栏「⚙ 生成本期下单计划」（全表动作，不在行内）→ 进入执行看板与终审（见 ⑦⑫）。

合计行（v13）

表底常驻一行合计：各周发货数量合计 + 下单数量合计（下单周 = 净额合计；非下单周 = W6 已排合计）。范围按角色分口径：销售 = 本人 SKU；主管 = 本部门；采购 / 终审 = 公司。口径标签显式挂在行上，且不随「待我处理」筛选变化——数量体感要稳定。金额仍只对终审 + 主管可见（2.9.5③），合计行只做数量。

角色透镜

同一张表、同一个 URL，四个角色（销售 / 主管 / 采购 / 终审）= 筛选视角 + 按钮组合。徽章数 = 状态机流到该角色的对象数。「待我处理」只显示需本角色处理的行，其余折叠；侧边栏「审核中心」已降级为「待办 (N)」深链接 = 打开本表 + 自动激活待我处理。

④ 审核状态机（一张表 · 审核 = 格子的状态）

不建独立审核中心装单据；每个修改对象（格子 / 净额）带一个状态，角色 = 谁有按钮。

无申请 → 🟡 待销售确认 → 🟠 待主管审（真实性）→ 🔵 待采购审（可执行性）→ [needFinal] 🟣 待终审 → ✅ 生效；任一环驳回 → ⛔ 回销售
状态（外框色）含义谁有操作按钮
🟡 黄待销售确认：系统重算例外格 / 前沿新周净额销售：确认无误（确认即承诺）/ 修改申请（→ 模拟）
🟠 橙待主管审真实性（活动是否真实、值不值这个量）。闸的算术系统已算好摆在卡上，主管只判事实主管：真实性通过 → 转采购 / 驳回（意见必填）；销售可撤回；采购可预加批注（随对象流转，不改顺序）
🔵 蓝待采购审可执行性（MOQ / 物流舱位 / 下厂交期）采购：可执行性通过（needFinal → 转终审；否则生效通知工厂）/ 驳回 / 加批注
🟣 紫待终审：已过前置，在「本期 PO 聚合」内随 PO 一并终审终审：聚合面板一键通过 / 个别调整
⛔ 红被驳回，回销售待办（驳回意见随单可见）销售：修改后再次提交 / 维持系统值确认
✅ 绿已生效（数值写回格子）—

时限与升级：周一销售确认 → 主管周三 18:00 前 → 采购周四 → 周五锁定（下单周周二先出 PO）。超时不自动放行（真实性审自动放行等于没审），超时项升级至 Ailah / 指定代审人并留痕 [升级对象待拍板]。

全程留痕：谁、何时、改了什么、理由——时间线收在抽屉底部，从早到晚，橙点 = 最新。

⑤ 销售确认与例外清单

销售的动作只有两类：常规行一键通过、例外逐项处理。逐行确认列已撤销。

例外清单（命中任一才要求逐格 / 逐项确认）

  • ① 系统重算动过的格（🟡 例外格）
  • ② 三态预警行（黄 / 红）
  • ③ 移仓 / 改派异常
  • ④ 新品期 SKU
  • ⑤ 前沿新长出的一周（确认对象 = 该周发货计划；下单额为只读结果，不需要销售确认）

一键通过：无在途例外的行确认即承诺；含例外的行提示逐项处理后生效。例外清单做成配置开关：前期拨「全量」，算法跑顺后拨「例外」，界面不改 [档位待拍板]。

⑥ 主管三档与批次签核

档位覆盖做什么
档一常规格（约 90%）零介入——审无法证伪的数 = 橡皮图章，不设按钮
档二修改申请逐条审真实性（🟠 状态，见 ④）。后期可按销售异议准确率切人级例外开关
档三每下单周期一次批次级签核专捕「该改没改」的反向遗漏（活动在即但销售未调数）——档二接不住的漏洞。永久保留

批次确认视图规则

  • 粒度：按团队一批；一条 deadline（非下单周 周三 18:00 / 下单周 周二 12:00，2026-07-07 拍板：PO 周二生成且以签核为前置，原统一 deadline 在下单周形同虚设）、一条升级链、一次签核。
  • 第一层 · 必看项：① 销售改过的格（可行内「真实性通过 / 打回」）② 落入活动窗口却未动数的格（活动来源 = 主管可维护的活动日历 [方案 B，待拍板]）③ 黄 / 红预警行 ④ 新品 ⑤ 移仓异常。
  • 第二层 · 扫一眼项：一键确认的常规格按销售人分组折叠，组头带首周合计 + 异议准确率（诊断价值留在视图层）。
  • 打回不阻塞：行内打回即踢出批次、回销售待办；批次照常可签。
  • 签核留痕 N/M/X：范围 N 个 SKU · 已逐条核 M 项 · 排除 X 项，显式写在按钮与留痕上——签的是什么范围、看过多少，白纸黑字。
  • 排除项去向：不进本期 PO，销售处理后走补充确认（次日 18:00 前补签，同升级链 [时限可配置]），量差由次周吸收（铁律②）。

⑦ 下单链与终审聚合

下单严、发货松（字典 2.2）：下单每 2 周、不可取消、上终审；发货每周、可调时间、不上终审（例外：红预警必断货、初期 W3+ 超线改量）。

下单链：销售 → 采购 → 终审，不经主管（字典 2.9）。终审的事实对象是 PO 总额而非格子，做成主表按 PO 聚合的模式切换（切终审视角出现聚合面板），不独立开发看板。

聚合面板字段

  • 需下单净额合计 / 涉及 SKU 数 / 金额（仅终审 + 主管可见，2.9.5 规则③）
  • 标红清单：needFinal 项（特殊申请超线、红预警空派、下单异议）+ 各自流转位置，可点开完整链路
  • 三段签核状态：主管批次 N/M/X · 采购 · 终审

生成下单计划（v10 新增动作）：下单周周二，采购点顶栏「⚙ 生成本期下单计划」——按当前净额冻结需求口径快照，推送「下单执行看板」（工厂 × 规格聚合、拆第 1/2 周、调拨在看板内计算），草案连同标红项转入终审。若仍有申请在主管 / 采购环流转，生成时提示"获批后需重生成"。

一键通过条件：主管批次已签核 ＋ 采购已生成 PO 草案 ＋ 全部标红项过完采购（进入 🟣 待终审位）。通过 = 标红项批量生效、数值写回、W6+W7 真锁、PO 不可取消（决策 19），锁信号回流表格，采购发合同下厂。终审聚合面板与下单执行看板是同一份 PO 数据的两个透视（金额视角 vs 工厂×规格视角）。

⑧ 趋势曲线与基础参数口径

曲线（≈3 个月 · 到货 & 安全库存天数）

  • 时间窗与空段压缩：约 3 个月；今天~首批到货前的空档（约 7/1~8/12 无到货）压缩到总宽约 16%，可调整区（后段）拿走 84%，轴上以 ≈ 断裂符标记。曲线含义：安全库存天数 = 可用（FBA + 在途）÷ 日均，逐日滚动；系统线（蓝）/ 销售线（黄）。
  • 节点 = 未来发货批次（W1–W7），仅未来：已到货节点不再显示——历史到货与「在库」含义重复，已并入曲线起点（起始天数由在库 + 在途快照推导）。节点位置 = 预计到货日（发货周 + 物流时效），详情直显在节点旁（周次 · 数量 / 到货日 · 时间调整标），免点击；悬停看完整信息：数量、渠道时效、到货 → 入仓 N 天 → 可售日（入仓时间已融合于此，原「基础参数」侧卡撤销）、该批天数增益、当天日均。
  • 快照信息行（常驻图上方）：当前在库（FBA 0:00 快照）· 在途（= 发货 − 已签收，已计入曲线起点）· 未交货订单 · 预计断货日（三态预警口径）· 入仓天数。
  • 全区悬停：图内任意位置对齐最近日期显示逐日天数，附跟随虚线；活动日底色（B3 排查口径）；图例在图表正下方。
  • 上下限：站点 × 淡旺季 × 新老品配置。跌破下限 = 断货风险；长期越过上限 = 备货偏多，砍最近批次。

⑨ 模拟演算规则（修改的唯一入口 = 图上拖节点）

  • 入口：展开趋势图，直接拖动批次节点。周格子只读展示（悬停提示指向图），格上仍显示紫色沙盘标记 800→600 同步反馈。抽屉里的「修改申请」按钮 = 展开趋势图开模拟。
  • 三个杠杆全在节点上：上下拖 = 数量（1 格 = 1 天库存 = 日均台数，按天吸附，下限 0）；左右拖 = 发货时间（按周吸附，可推迟 ≤2 周 / 可提前 ≤2 周，出货日不得早于今天、到货点须留在窗内）；点渠道名 = 循环切换（空派 7 / 快线 30 / 普线 35 / 慢船 45）。
  • 点 / 拖分流（阈值 6px）：点节点 = 弹详情面板（数量 / 时间 / 渠道下拉 / 到货→可售 / 影响天数），改渠道即时重算基准线；拖节点 = 改量改期。拖动 <6px 视为点击，消除"一点就掉"。
  • 拖后确认机制：拖到新位置后弹「✓ 确认 / ↩ 还原」气泡就地显示；未确认即自动还原（关闭面板 = 放弃），确认后气泡变「→ 转为修改申请」——就地提交，不必回顶栏。
  • 权限继承周段闸：W1–W2 节点橙描边（守工厂节奏，v17）——销售可提前 / 加量 / 改渠道（走主管链），不可推迟 / 减量；采购可推迟（减压）/ 提前 / 加量 / 改渠道（🔓 即时生效留痕），不可减量。W3+ 提前不再钳在锁定区边界（提前=救断货，允许），但到货点不得早于今天 / 不得拖出窗外。PO 落地后 W6/W7 节点禁拖。
  • 警示实时：拖动过程中模拟虚线、节点标注、状态条结论（超上限 X→X′ · 最低 Y→Y′ · 跌破下限 ⚠）逐帧刷新；拖了个寂寞（回到原值）不算改动。
  • 纯沙盘：不落数据，重置即清空；转为修改申请时模拟证据自动带入（只读）＋ 补充说明必填，理由格式 【模拟证据】…【补充说明】…；多周改动捆绑提交、每周各走各的闸（③ 矩阵），不合规的周被拦并列明原因。
原型曲线与拖拽换算为等比 mock；真实 ERP 中模拟计算走 2.5.2 滚动模拟重算全链（B1 负库存不 floor 为前提）。

⑫ 与采购执行看板的联动契约

「下单执行看板」（工厂 × 型号 × 规格）与「发货执行看板」（工厂 × SKU × 滚动周）是发货计划演变同一事实层的不同透视：主键不同（工厂 / 规格 vs 站点 / SKU）、用户不同（采购 / 工厂 vs 销售）、生命周期不同（PO 生成后 vs 生成前），独立成板块、不并入本表。联动规则四条：

#规则内容
1单一事实源周格数量的唯一产生地 = 发货计划演变（销售审批链 + 采购 W1–W2 应急直改）。执行看板对数量只读、不重算。
2下行（需求 → 执行）净额按工厂 × 规格聚合 → 下单执行看板「本期应下单」；跨站点调拨 / 在产让单在看板内抵扣（可调对象 = 同工厂、在产未交付、规格可改派——三个判断条件只在工厂维度可见，这是调拨必须落在执行看板的原因），得「实际新下单」并拆第 1/2 周。
3上行（唯一写回）发货执行看板上采购做的时间前挪 / 后移、渠道调整（发货松，采购职权）写回发货计划数据层，留痕 + 销售知会；数量永远不从执行看板改。
4锁信号采购「生成下单计划」→ 执行看板成形 → 终审通过 → 锁信号回流，发货计划页 W6+W7 真锁（决策 19）。

双口径定义（防对不上数）
