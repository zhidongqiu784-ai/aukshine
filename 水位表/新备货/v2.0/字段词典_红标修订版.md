## 2.0 参数与逻辑分离(设计总则)

### 2.0.1 原则:算法跟数据源联动,不依赖固定值

所有影响备货的参数(交期、时效、安全库存天数、系数、活动日期等)未来都会随实际情况变化。因此整套设计遵循:

- **算法只引用参数名,不写死数值**。例:预计可售时间 = 今天 + 产品交期 + 海运时效 + 入仓时间(这几项都是可配置参数,不写成"今天 + 45 + 37 + 7")。
- **参数集中在 AOK ERP 配置中心作为数据源**。改参数只改配置中心一处,所有引用该参数的算法自动用新值,**不用改算法逻辑**。
- **后续所有 ERP 板块以配置中心的参数为基础数据源**。设计时引用参数,不复制固定值,避免口径分叉。

> 本章及全文档出现的具体数值(如 45 天)均为**当前配置值的示例,仅供理解参考、不作为算法依据**;算法实际运行时从配置中心实时取值。数值变更只改配置中心 + 下方参数总表,不改算法描述。

### 2.1.1 字段父子关系图

```
                          总库存
                             │

        ▼              ▼           ▼              ▼
    FBA 库存       海外仓库存     在途库存      未交付订单

        ├ 待调仓        └ 全新品(海外仓全新)
```

**关键关系**:
- **FBA 可售 ⊂ FBA 库存**(可售是库存的子集,**不能重复加**)

- **未交付订单不进总库存**(还在工厂)

### 2.1.3 库存的"到 FBA 可售时间"差异

不同库存位置到"亚马逊上架可售"所需的时间不同:

| 库存位置 | 到 FBA 可售时间 |
|---|---|
| **FBA 可售** | 立即(已经可售) |
| **FBA 库存(可售之外的部分)** | 几天内(亚马逊内部处理,不需手动触发) |
| **海外仓库存(默认在途)** | **7 天调拨 + 该站点入仓时间(随到货时淡/旺季变)** |
| **在途库存** | **剩余海运/空运时间 + 该站点入仓时间** |

**为什么海外仓算"默认在途"?**

海外仓库存物理上虽然不在路上,但只要每周一系统判定"需要补货",会自动形成调拨指令,7 天调拨 + 入仓时间后到达 FBA 可售状态。所以从算法的视角:**海外仓库存 = 一种"起点是今天、固定 7 天调拨 + 入仓时间到货"的特殊在途**。

**重要区分:两种判定使用不同口径**(详见 2.4 三态预警 和 2.7 调拨指令机制):
- **三态预警(客观预测)**:海外仓默认在途要算入 FBA 可售模拟
- **调拨触发(主动决策)**:海外仓默认在途**不**算入 FBA 可售模拟(否则永远不会触发)

> **保守估计注意**:海外仓默认在途的到货时间(7 天调拨 + 入仓)是乐观假设。若实际遇清关延误 / 仓位限制,海外仓的货会晚于此到 FBA 可售。各算法(本期下单 / 三态预警 / 调拨)用海外仓默认在途做滚动模拟时,应按配置中心维护的保守到货时间算入,避免高估未来库存、少下单而埋下断货风险。

### 2.1.4 双口径设计的由来

**为什么 FBA 可售 和 FBA 库存 要分两个口径?**

这是从一次实际案例复盘得出的经验:
- 销售看到"FBA 库存 1000 台",以为可以发货上活动
- 实际可发的"FBA 可售"只有 700 台,其余 300 台在调仓中
- 活动启动时实际可发货太少,被迫延期

**所以**:
- **断货预测和滚动模拟必须用 FBA 可售**(短期视角,只看真正能立刻发的)
- **库销比公式用总库存**(中长期视角,海外仓和在途库存迟早会到 FBA)

### 2.1.5 总库存的精确口径(对齐现有系统,2026-06-02 拍板)

库销比、滞销判定用的"总库存",**口径以现有 NocoBase 生产系统为准**:

> **总库存 = 在途 + FBA 可售 + FBA 待调仓 + 海外仓全新品 + 海外仓检测品**

- **与 2.1.2"FBA 库存全口径"的差异**:现有系统总库存的 FBA 部分**只算 FBA 可售 + 待调仓**,**不含"调仓中、入库中"**——这两块是 FBA 库存里暂时锁定 / 未上架的部分,现有系统不计入总库存。
- **影响**:总库存略偏保守(少了调仓中 / 入库中),库销比相应略偏小(偏短缺方向)。这是现有系统现状,已确认接受。
- **库销比** = 总库存(上述口径)÷(加权基准日均 × 30)

> **IT 落地确认点**:现有库销比分子的实际取数字段须对齐到本口径;若发现取的是别的库存口径,以本节"总库存"定义为准修正。

### 2.1.6 SKU 成本单价(2026-06-10 新增,经营分析金额维度)

**背景**:终审看板"经营三问"(钱压在哪 / 要亏多少 / 系统比人准吗)需要把台数换算成金额(本期下单占用资金、滞销占压、断货预计损失、空派成本 vs 救回营收),而**当前全系统只有台数、没有金额维度**——这是经营三问能落地的唯一新增字段。

| **用途** | **仅经营分析金额测算**(终审看板经营三问区):库存占用金额 = 库存台数 × 成本单价;滞销占压 / 断货损失同理 |

### 2.2.2 加权基准日均的算法

```
原始日销量 → 除以 当日的日类型系数 → 得到归一化日均
30 天的归一化日均 → 用加权权重平均 → 加权基准日均
```

**关键**:
- 加权基准日均不是简单的"过去 30 天总和 ÷ 30"
- 它会**剥离活动日的异常**(用日类型系数除一下)
- 它会**给近期更高权重**(近期销量更能反映当前趋势)

> ⚠ **2026-06-10 补(D2 根因,IT 确认项)**:当前定义只剥活动日、**不剥断货日**——断货日销量 0 不是真实需求 0,是无货可卖;计入分母会压低加权基准 → 库销比虚高 → 误标滞销(2.5.6 排除态只能止血在"断货中 / 恢复期",修复后 30 天窗口内仍虚高)。**修复口径:断货日(FBA 可售 = 0 的天)不计入 30 天加权分母**,做法同日类型剥离。⚠ 先查 NocoBase 现有系统 `daily_sales` 加权基准是否已这么算:已剥则只补文档,未剥则进 IT 落地清单。**只剥断货日(库存 0),不剥"销量低但有货"的天**(真实疲软该进基准)。

### 2.2.3 与"每日预估销量数组"的区别

| 维度 | 加权基准日均 | 每日预估销量数组 |
|---|---|---|
| 视角 | 过去 30 天(历史) | 未来 90~180 天 |
| 形式 | 一个数字 | 一个数组(每天一个值) |
| 用途 | 算安全库存(乘以安全天数) | ~~算覆盖期销量(累加 14 天)~~ <font color="red">算发货周服务期需求(逐周累加 7 天)与下单批 14 天需求(2.5.2 修订)</font> |
| 是否含活动 | 不含(已剥离) | 含(数组里 BD/PD 当天的值会更高) |

**为什么需要两个?** 安全库存是保底用的,要稳定基准;覆盖期销量是预测用的,要反映未来活动的影响。

### 2.3.2 关键边界规则

| 规则 | 说明 |
|---|---|
| **今天的时区** | 各站点取当地 0:00 起算 |
| **W 周边界** | 周一 0:00 切换。今天是周一 0:00 之后,W1 = 下周一开始那一周 |
| **海运/空运起算点** | 从工厂出货当天起算(不是装柜日,也不是签合同日) |
| **入仓时间** | 货件到亚马逊仓库 → 上架变为可售 之间的天数 |
| **海外仓调拨时效** | 7 天(所有站点统一) |
| **预计断货日定义** | 用 FBA 可售口径滚动模拟得出 = FBA 可售首次为 0 的那天 |
| **节假日** | 中国节假日(春节)影响工厂交期;美国节假日(感恩节)影响入仓时间。具体节假日在配置中心维护 |
| **跨日处理** | 算法每天 0:00 跑,跑出来的结果在~~次日~~<font color="red">**当天** 9:00 之前(2026-07-07 修正,理由见 2.0.3 算法运行节奏)</font>提供给销售 |

### 2.3.3 W 周次的完整定义

| 编号 | 含义 | 销售可调? |
|---|---|---|
| **W0(本周)** | 已发已锁,**不进 W 编号体系**(只是历史参考) | 不可调 |
| **W1(下周)** | 工厂已排产 | ~~不可直接调~~ <font color="red">**守工厂节奏(2026-07-07 拍板)**:销售不可推迟 / 减量;可提前 / 加量 / 改渠道(转申请走主管链);采购可推迟(库存堆积减压,克制使用)/ 提前 / 加量 / 改渠道(应急直改留痕),不可减量。详见 2.4.5</font> |
| **W2(下下周)** | 工厂在排料 | <font color="red">同 W1(守工厂节奏)</font> |
| **W3 起** | 后续 | **两条通道**:发货**时间顺延**归系统(每周一跑,2.5.3);发货**量**可由销售手动调<font color="red">——决策 27:安全区间闸,区间内免审自动通过、出界走主管 + 采购部(2.5.7 修订)</font> |

**为什么 W1 = 下周不是本周?** 因为本周的货已经发出去了,改不动。把它叫 W1 没意义,销售看了也调不了。所以系统把"本周已发"剥离出 W 编号体系,W1 直接从下周开始算——这样销售看到的所有 W 都是"还能改的"。

<font color="red">**前沿逐周滚动(2026-07-07 补,决策 25)**:发货计划每周一在前沿长出一格新周(非下单周长出下单批首周、下单周长出次周),销售确认即承诺;下单批 = 预计可售映射到的两周(2.3.4),下单周合并两周出净额 PO(2.5.2)。天然不重不漏。**W8+ 只读预览区(2026-07-07 补)**:发货计划表尾提供 W8+ 系统模拟预览(只读、不确认、不承诺、不进审批)——反转后销售视线收窄到 W7,旧覆盖期公式承载的中长期视野(旺季前产能预沟通)由此承接;样板 UI 待实现,列待办。</font>

### 2.3.4 预计可售时间 → W 周次映射(算法内部用,UI 不展示 W)

把"本期下单的预计可售时间"换算成具体 W 周次,供「发货计划演变」横轴高亮用。**纯算法内部口径,界面统一显示日期范围(如 8/4-8/17)、不显示 W 编号**。

- **下周一起始日** = 今天 + 距下个周一的天数(本周已锁,不进 W 体系)
- **落在第几个 W** = 向下取整((预计可售时间 − 下周一起始日) ÷ 7) + 1
- 本期下单覆盖两周 → 返回 {首周 W, 首周 W + 1}<font color="red">——此两周即发货计划的**下单批**;PO(净额)= 两周发货计划合计 − 未交货订单余量(2.5.2 修订,2026-07-07)</font>

> 不同站点交期 / 时效不同,同一天下单的预计可售时间能差 3-5 周,必须按 today 实时算,不写死。

### 2.4.2 预警判定规则

```
情况 1:预计断货日 ≥ 海运到货可售日       → 🟢 绿(无预警)
情况 2:海运到货可售日 > 预计断货日 ≥ 空派到货可售日  → 🟡 黄(改运输方式可解决)
情况 3:预计断货日 < 空派到货可售日       → 🔴 红(必断货)
```

### 2.4.3 三态预警含义与处置

| 等级 | 含义 | 处置在哪 | 谁发起 | 是否需要终审 |
|---|---|---|---|---|
| 🟢 绿 | 海运到货前不会断货,正常下单即可 | (无,继续走常规流程) | - | - |
| 🟡 黄 | 海运赶不上但空派可救 | 「发货计划演变」对应 W 格子标黄 | 系统自动出理想出货计划 → 销售/采购部沟通工厂 | **不需要**(只改运输方式不改总量,风险低) |
| 🔴 红 | 空派也来不及救,必断货 | 「⚠️ 断货特殊处理」侧边栏板块 | 系统自动出理想出货计划 → 销售/采购部沟通工厂 | **需要**(涉及大额空运成本和发货策略) |

### 2.4.4 黄/红预警两阶段流程

```
阶段 1:理想出货计划
  ↓
  系统输出 = 出货计划(也是工厂的交货计划):
    | 日期 | 渠道 | 数量 |
    | 5/15 | 空派 | 200 |
    | 5/22 | 海运 | 300 |
  ↓
  销售/采购部拿这份去跟工厂沟通

阶段 2:实际出货计划
  ↓
  采购部填工厂实际确认明细:
    | 日期 | 数量 |   (不分渠道,只填工厂能交什么)
    | 5/15 | 150 |
    | 5/22 | 200 |
  ↓
  系统基于工厂实际能力,重算最优空派/海运组合
  优化目标:尽可能少发空派(降低成本)
  ↓
  输出实际出货计划 + 真实空运成本
  ↓
  红预警:销售确认 → 终审 → 执行
  黄预警:销售确认 → 采购部审核通过 → 执行
```

**关键约定**:
- 黄预警和红预警共用两阶段流程(逻辑统一)
- 终审只在**红预警 + 阶段 2 之后**(看的是真实成本和商业决策)
- 阶段 1 不需要终审(还没有真实成本数字)

### 2.4.5 W1/W2 锁定机制

**目的**:避免对工厂频繁微调发货量。

**核心规则**:复用预警状态判断,零额外计算。

```
系统每周一跑算法,W1/W2 算的应发量与原计划有差异时:
  SKU 状态为绿:拦截调整,维持原计划(不调,不通知工厂)
  SKU 状态为黄:走黄预警两阶段流程
  SKU 状态为红:走红预警两阶段流程
```

<font color="red">**人工调整权限矩阵(2026-07-07 拍板,新增;守的点 = 工厂节奏:工厂已排产、货不能堆在工厂)**:

| 动作 | 销售 | 采购 |
|---|---|---|
| 推迟(往后挪) | ⛔(销售若可推会全线调,打乱工厂计划) | ✅ 直改留痕(仅限库存堆积减压,克制使用;采购只调刀刃上的) |
| 提前 | ✅ 转申请走主管链(断货救援) | ✅ 直改留痕 |
| 加量 | ✅ 转申请走主管链 | ✅ 直改留痕 |
| 减量 | ⛔(货已生产,少发只堆仓;减压走采购推迟 / W3+ 顺延) | ⛔ |
| 改渠道 | ✅ 转申请走主管链 | ✅ 直改留痕 |

**与安全区间闸(决策 27)的边界**:区间闸**不适用于 W1/W2**——此处审的是货源 / 舱位**可执行性**(中国仓有没有现货、订不订得到舱),不是库存风险,库存后果再健康也替代不了这道核对;守工厂节奏的方向限制(禁推迟 / 减量)同样维持。若会议本意是 W1/W2 也放开,需单独拍板。</font>

### 2.4.6 28 天恢复期

**触发条件**:实际断货 > 7 天的 SKU,接上后自动启动。

```
接上当天起前 14 天:每日预估销量 × 0.7
后 14 天:每日预估销量 × 0.85
28 天后:不修正,系统用实际销量数据自然驱动,自动解除
```

**不触发**:断货 ≤ 7 天的 SKU 接上后直接用每日预估销量数组,不修正。

### 2.4.7 恢复期与黄/红预警的关系(决策 18)

**不冲突,顺序处理**——不需要为预警单独写规则。

```
正常 → 断货 → 接上(启动恢复期)→ 恢复期内(销量打折)→ 恢复期外(正常)
        ↓
        断货期间:暂停预警(销量为 0,跑预警无意义)
        接上后:
          - 恢复期内:本期下单 / 发货 / 三态预警 / 调拨触发 都用打折销量(× 0.7 / × 0.85)
          - 恢复期外:用真实销量,正常跑
```

**关键**:所有"基于每日预估销量数组"的算法,在恢复期内都自动用打折值。因为打折是改销量数组本身,所有读这个数组的算法(下单 / 发货 / 预警 / 调拨)自动跟随,**不需要为预警单独写规则**。

**恢复期的核心用途**:给"接上后重新跑下单 / 发货算法"用,解决断货后销量数据不稳定时怎么算备货量。预警、调拨跟随打折销量只是自动效果,不是设计目的。(折扣系数 前 14 天 ×0.7、后 14 天 ×0.85 的定义见 2.4.6)

**恢复期状态显示字段**:

| 字段 | 取值 | 说明 |
|---|---|---|
| **断货状态** | 断货中 / 恢复期 / 正常 | 断货中暂停预警;恢复期、正常都正常跑预警 |
| **恢复期进度** | 恢复期 X/28 天 | 接上后显示,28 天后自动解除标识 |
| **恢复期销量折扣** | 前 14 天 ×0.7 / 后 14 天 ×0.85 | 当前生效的折扣系数 |

### 2.4.8 黄预警 · 空派数量计算(函数 I)

承 2.4.4 黄预警阶段 1:SKU 状态为黄时,系统**自动算需改空派的台数**(不需销售输入,销售可改)。

- **缺口期** = 预计断货日 → 海运到货可售日(这段海运补不上,需空派填)
- **缺口期销量** = 每日预估销量数组在缺口期内的合计
- **空派缓冲量** = 加权基准日均 × 空派缓冲天数(防海运延迟致空派量刚好不够;缓冲天数见 2.0.2,当前 7 天)
- **空派数量** = 缺口期销量 + 空派缓冲量
- 同时给出**海运 / 空派拆分**:空派数量(缺口期算出,**可能跨多个发货周**)**按出厂日从早到晚逐笔扣减到各发货批次**——优先把最早能发的批次改空派,直到空派数量分配完(**每笔空派只分配一次,某周分不完的剩余滚入下一批**);每个发货周"仍走海运台数 = max(0, 该周原计划量 − 已分配到本周的空派量)"

审批:销售申请改运输方式 → 采购部审核,**不上终审**(2.9)。

> **实现状态(2026-06-02 数据校准)**:核对 NocoBase 生产系统,无任何空派相关表——本算法属 **V3 新建功能、现有系统未实现**。逻辑合理,实现时需用真实黄预警案例验证缓冲天数等参数。

### 2.4.9 红预警 · 空派最优解测算(函数 J)

承 2.4.4 红预警阶段 2:SKU 状态为红时,在「断货特殊处理」板块由销售触发。
**销售输入**:工厂可分批交货明细(出厂日 + 数量)、空运 vs 海运每台成本差。

- **优化目标**:红预警 = 空派也来不及、**必断**,所以目标是**最小化断货天数 / 损失台数**(不是"消除断货"——那做不到),在此前提下尽量少发空派(省成本)
- **算法口径**:把工厂分批按出厂日从早到晚累加,每加一批就模拟"已累加批次走空派 + 剩余走海运"的库存曲线,**取使断货天数 / 损失台数最小的累加组合**(若恰好存在让全程不破 0 的组合,即取它、断货归零——但此时已非"必断",退化为黄预警情形)
- **无分批可加**(工厂无加急)→ 全程海运方案
- **全部累加仍不够** → 接受断货方案,输出断货天数 + 损失台数
- 输出字段:空派批次 / 空派数量 / 额外空运成本(= 空派量 × 成本差)/ 海运量 / 断货天数 / 损失台数(字段定义见 2.4.1)
- **不算总成本 / 毛利**——只给空运额外成本 + 损失台数,商业账由终审评估

审批:红预警阶段 2 → 销售确认 → **终审**(涉大额空运成本,2.9)。

> **实现状态(2026-06-02 数据校准)**:同 2.4.8,空派最优解属 **V3 新建功能、现有系统未实现**,待真实红预警案例验证累加法与成本口径。

### 2.4.10 链接异常冻结(2026-06-10 补编号,口径从业务逻辑 §3.11 收编为单一源)

**触发**:链接状态 = 下架 / 暂停 / 审核中(监听层,需领星 API,见系统设计板块 25)。

**冻结动作(立即,系统自动)**:① 冻结该 ASIN **全部未发货计划**;② 库存标"冻结中"、不计入可用;③ 即时通知销售 + 采购部;④ 进"状态异常清单"每日复核。

**优先级**:**高于 W1/W2 锁定**(链接下架时已排产订单也要冻结);动作层为最高优先级——冻结期间下单 / 调拨 / 空派 / 顺延 / 紧急追加**全停**,唯改派让单放行(订单让走反而止损),见 2.5.8 仲裁矩阵。

**解除**:链接状态恢复 → 自动解冻、计划恢复滚动。留痕:冻结 / 解冻时点 + 状态来源。

**为什么**:链接异常时系统还按正常销量预估,会积压或误判断货——机制兜住"产品没法卖了但系统不知道"的盲区。

- 余量为中间值 → 精确补足缺口。</font>

> - **分子(总库存)精确口径见 2.1.5**(对齐现有系统:在途+FBA可售+待调仓+海外仓全新+检测,不含调仓中/入库中);公式块里"(FBA库存+海外仓+在途)"是示意展开,精确以 2.1.5 为准。

### 2.5.3 W3+ 顺延机制

**机制目的**:处理"销量比预期低/库存堆积"场景。从 W3 开始,发货**时间**的顺延由系统统一处理(销售不手动改"顺延"这个动作)。<font color="red">与净额链路的关系(2026-07-07):顺延改的是发货计划周的时间,净额随发货计划自动重算,无需为顺延单独写下单规则;W1/W2 的人工推迟通道(采购,2.4.5 矩阵)与本节系统顺延并存、互不替代。</font>

> **与销售手动调发货量的边界(2026-06-05 对齐,原 2.3.3/2.5.3"不由销售直接调"措辞过宽已收窄)**:W3+ 有**两条并存通道**——① 本节的**系统顺延**:改发货**时间**、自动、只对绿色 SKU 库存堆积;② 2.5.7 的**销售手动调发货量**:改**数量**、销售主动、走三道门槛。**销售手动调量后,系统须重跑本节 W3+ 顺延模拟**,否则顺延依据的库销比已失效(**触发顺序:以销售调量确认为触发、重跑在调量之后**,避免自动顺延先跑、用旧销量数组覆盖调量结果)。

**判定逻辑**(无需阈值):
```
系统每周一跑 W3+ 检查
  ↓
对每个 绿色 SKU 模拟:顺延 1 周后,库销比是否更接近健康区间?
  ├── 是 → 触发顺延
  └── 否 → 维持原计划
```

**顺延流程**:
```
触发顺延
  ↓
默认动作:整批顺延 1 周(W3 → W4,W4 → W5,以此类推)
  ↓
内部自检:用 FBA 可售 + 海外仓到货 + 在途到货 滚动模拟,顺延后会断货吗?
  ├── 不会断货 → 整批顺延 1 周
  └── 会断货 → 部分保留(够撑到下批到货) + 剩余顺延
              算"必须保留多少才不断货"留在 W3
              剩下的整批顺延 1 周
  ↓
推送给采购部审核(无需终审)
  ↓
通知工厂
```

**累计顺延提醒**:
- 单次顺延不上限,逐周自然评估
- 累计顺延超过 4 周时,系统提示销售/采购部:"该 SKU 已累计顺延 N 周,要不要重新评估销量预测?"
- 这是**提醒,非强制阻止**

**预警优先**:
- 黄/红 SKU 不进入顺延流程(预警机制独立处理)
- 只有绿色 SKU 才走顺延逻辑

### 2.5.4 产品标签与淘汰预警处理(决策 21)

**产品标签**是产品在库存水位表里的生命周期 / 处置状态(ERP 现有字段,本节定义其取值与备货算法解读)。一列承载两类信息:**前三段是按上架天数自动算的"年龄",淘汰期是采购部确认淘汰后人工打的"处置态"(优先级最高,盖掉年龄)**。

|---|---|---|
| **新品期** | 自动:上架 < 90 天 | ~~用新品安全库存天数~~ <font color="red">安全库存统一 7/14,无新品档(决策 28)</font>;开售前走新品首批备货逻辑(2.11) |

**淘汰预警触发逻辑**:**淘汰预警**是决策 21 新建的预警,区别于库存板块已有的「滞销预警」(库销比 > 上限,纯库存信号)。淘汰预警 = 滞销 + 利润不达标同时成立——只滞销但利润率仍达标(≥ 10%)的产品多半只是短期备多、本质健康,不亮淘汰预警(注:利润率 < 10% ≠ 亏损,只是利润偏低)。

```
淘汰预警触发(以下全满足,AND;按 ASIN 判定):
  ① 库销比 > 4.5                     (即"滞销",全局固定,决策 28;口径见 2.5.2)
  ② 连续 2 个月 月利润率 < 10%       (利润维度;数据从 ERP 销售数据调取,门槛 10%/连续月数为配置参数)
  ③ 非"断货中 / 恢复期"状态         (这两态月销量被压低、库销比虚高,排除以免把断货误判成滞销)
→ 亮淘汰预警 → 推送采购部
```

> **不分新老品**:新品期同样纳入——新品第 2 个月利润率也要 ≥ 10%,连续 2 个月 < 10%(即上架满 2 个月仍达不到)同样触发淘汰预警。这样上架即做废的新品也有止损入口(不再靠"新品期除外")。注:这里"不分新老品"只针对**淘汰预警触发**;~~备货算法(安全库存两档、新品首批备货)该分仍分。~~<font color="red">决策 28 后安全库存不再分新老品(统一 7/14);新品首批备货**流程**仍独立(2.11)。</font>

### 2.5.5 异常销量识别与"其他异常预警"(决策 22)

异常销量(尤其无活动标记的突发高销量)会污染加权基准日均(→ 安全库存 / 覆盖期销量虚高、备货过多)和月销量 / 库销比(→ 库销比虚低、掩盖淘汰预警)。决策 22 分两块处理:

**A. 蹭 BD(绑定变体蹭活动)—— 人工补标**

A 产品自己没报活动,但与报了 deal 的 B 产品绑定变体,跟着出了活动销量;系统识别不到 A 有活动,不会用日类型系数剥离这段。**这种情况少见,不做自动识别**:销售发现后反馈采购部 → 采购部按销售反馈的实际活动类型,给 A 那几天手动补打日类型 → 系统自动用日类型系数剥离(机制见 2.2)。**手动补标留痕**(操作人 / 日期 / 调整的日期范围与类型)。

**B. 销量突增 →"其他异常预警"**

系统自动检测异常突增,亮"其他异常预警"(提示性预警,与淘汰预警、三态预警并列,归类"其他异常")。

```
触发(全满足,按 ASIN;倍数 / 门槛 / 天数均为配置参数):
  ① 连续 3 天 实际销量 > 加权基准日均 × 1.5
  ② 该 ASIN 日均 ≥ 10 单        (绝对量门槛,避免小销量品天然波动频繁误报)
→ 亮"其他异常预警" → 推采购部研判:
   ├ 未被识别的活动(如绑定变体蹭量) → 走 A,手动补标
   └ 真实销量上涨               → 关注备货,交三态预警 / 覆盖缺口加单(决策 20)
```

> **为什么对照"加权基准日均"而非"前一天"**:对照前一天易被单日低谷虚报,且"跳台阶后维持高位"会漏报;对照加权基准(常态线、已剥离已知活动)能稳定捕捉"持续高于常态"。
> **B 是 A 的自动发现入口**:突增检测把蹭量自动推给采购部,补上"蹭 BD 纯靠销售眼尖"的软肋。

性质:提示性预警,本身**不直接驱动备货动作**。**闭环**:亮预警 → 采购部研判 → 判**真起量** → 采购部确认后**调高预估销量数组 / 触发覆盖缺口加单**(走 2.5.2),避免"看见了没改单"导致断货;判**蹭量** → 走上方 A 手动补标。

> **待细化**(Codex 校验补,后议):① 加单闭环已补入上方性质段(采购部确认后调预估 / 加单);②  蹭量未补标会抬高加权基准、削弱突增检测灵敏度(自我掩盖),突增检测可同时对照预估销量或用抗污染基准;③ 蹭 BD 事后补标只影响后续下单(已下单不可取消,决策 19),不回溯改已生成采购;④ 其他异常预警 / 三态 / 淘汰预警同时亮的优先级原则(实际罕见,暴涨会拉低库销比、与滞销互斥)。

### 2.5.6 滞销判定与库龄定位(原函数 F,经 NocoBase 数据校准撤回四级模型)

> **校准说明(2026-06-02)**:旧算法 SOP 函数 F 设计了"老货占比 + 售出率 → 极高/高/需关注/正常"四级风险模型,但核对生产系统(NocoBase「滞销清单」SQL 视图)后确认**该模型从未被采用**。现有系统的真实做法是:**滞销判定看库销比、库龄数据用于定位滞销集中在哪些店铺**。故撤回四级模型,改为对齐现有做法,避免造一套与库销比并行打架的冗余算法。

**滞销判定(归库销比,见 2.5.2)**:
- 库销比 = 总库存 ÷(加权基准日均 × 30),即"按当前日常速度还能卖几个月"
- **滞销 = 库销比 > 4.5**~~;上限 = 完整备货周期最大值 ÷ 30(动态)~~ <font color="red">(全局固定,决策 28;不再随站点时效变)</font>
- **排除态(2026-06-10 补,对齐 2.5.4③)**:**断货中 / 恢复期不判滞销**——这两态月销被压低、库销比虚高,不排除会把断货品误标滞销,进而误触发改派让单(2.7.6)/ 误入滞销清单,雪上加霜。
- ⚠ **根因残留提示**:断货修复后(已不在排除态)30 天窗口内库销比仍可能虚高(加权基准含断货 0 销量天),根因修复见 2.2.2 批注(剥离断货日,IT 确认项)。
- 现有生产系统用固定阈值"库销比 ≥ 5 月"判滞销;V3 改用上述**动态阈值**(自适应站点),固定 5 月作经验校验值保留参考

**滞销态 × 下单建议并存的复核规则(2026-06-10 补,D3 收编 HTML 现行逻辑)**:滞销看**存量现状**、下单算**未来流量**(覆盖期在约 90 天后),时间视角正交、可同时为真(典型:淡季滞销表象 + 旺季覆盖期真需求)。规则:
1. 滞销态 + 本期下单建议 > 0 → **不加硬闸**,标"⚠复核",**强制人工复核后才可确认**;
2. 复核三查:① 覆盖期销量来源(活动日贡献占比)② 预估数组是否被突增 / 蹭量污染(联动 2.5.5)③ 当前季节(淡滞销 + 旺覆盖期 = 典型合理共存);
3. 复核留痕(谁核 / 结论 / 日期)。与 2.5.2"自然不下单"的关系:那是缺口 ≤ 0 的常态;本规则管缺口 > 0 的例外共存。动作层取舍见 2.5.8 仲裁矩阵。

**库龄的用途(归「滞销清单」板块,不做独立算法)**:
- 库龄分段(91-180 / 181-270 / 271-365 / 365+ 天)用于**在已判滞销的 SKU 内,定位库存压在哪些店铺**
- 店铺排序权重 = 该店铺(181-270 + 271-365 + 365+ 天)滞销库存合计,从重到轻排,展示前 N 个店铺
- 这是滞销清单板块的**展示逻辑**,不是字典算法;库龄**不参与**"滞销是否成立"的判定

> **若将来要独立的"积压风险等级"**(库龄维度单独分级):属增量优化,届时再立,不在 V3 首版。当前库龄只做定位 + 排序。

**适用边界**:本表管"同 SKU 多状态共存"的动作取舍;**不管**单机制内部流程(各机制各自已有)、不替代 2.5.6 的"滞销 × 下单"专项复核条款(那管缺口数值层,本表管动作开关层)。各机制引用本表只引编号,不复述格子。

## 2.6 字段在算法各处的使用归属(防止用错口径)

| 算法/场景 | 用什么字段/口径 | 为什么 |
|---|---|---|
| **库销比公式** | 总库存 ÷ 月销量 | 中长期视角,海外仓和在途库存迟早可售 |
| **健康度判定**(短缺/正常/滞销) | 库销比(总库存口径) | 与库销比公式同口径 |

## 2.7 调拨指令机制(海外仓 → FBA)

### 2.7.1 设计动机

海外仓库存如果只是"放着",FBA 缺货时也不会自动到 FBA。所以系统需要主动判断"什么时候该让海外仓库存动起来",形成调拨指令推送给采购部。

**核心理念**:海外仓库存默认"准备调拨",但只在算法判定"FBA 真的会缺货"时才形成实际指令——避免无谓调拨,也避免漏调拨。

### 2.7.2 调拨指令的生命周期

```
海外仓库存(待命状态,算法上默认为在途)
    ↓ 系统每周一判定 → 需要调拨
    ↓ 形成调拨指令 → 推送给采购部审核
    ↓ 审核通过 → 调拨指令生效
海外仓在途(7 天调拨 + 入仓时间到 FBA)
    ↓ 到货 + 入仓完成
FBA 可售
```

### 2.7.3 调拨触发判定

**频次**:每周一系统跑(跟发货计算同频)

**判定窗口**:
- **淡季(常年):14 天**
- **旺季(根据 AOK ERP 淡旺季配置):21 天**
- **季节锚点(2026-06-10 补明)**:窗口档位按**判定日(跑批日)当前季**取——窗口是"现在往前看的提前量",与今天所处季节相关;若按预计到货日取会引入循环依赖(窗口决定到货评估范围、到货又决定窗口)。全局锚点归口见 2.0.2 季节锚点归口表。

理由:旺季销量天花板高 + 入仓时间长(美国/德国旺季入仓 14 天),需要更长的提前量;淡季销量稳定 + 入仓时间短(5-7 天),14 天窗口足够。

**判定逻辑**:
```
对每个 SKU:
  第一步: 滚动模拟未来 [窗口] 天 FBA 可售
          起始 = 当前 FBA 可售
          加 = 国内在途到货(在窗口内到的)
          减 = 窗口内预估销量
          注意:这里"不算"海外仓默认在途 (主动决策视角)
  
  第二步: 窗口内 FBA 可售有为 0 的天吗?
          ├── 没有 → 不调拨(海外仓库存维持待命)
          └── 有 → 触发调拨
                    ↓
                    调拨量 = 让"窗口内 FBA 可售刚好不为 0"的最少量
                    约束:≤ 当前海外仓库存
```

**多 SKU 同时触发**:按 FBA 可售见底时间早晚排序

**审批流**:采购部审核 → **不需要终审**(类似顺延机制,无大额成本)

### 2.7.4 关键双口径(防止用错)

| 判定 | 视角 | 海外仓默认在途算入吗? |
|---|---|---|
| **三态预警** | 客观预测(会不会断货) | **算入**(因为会调拨,客观上会到 FBA) |
| **调拨触发判定** | 主动决策(要不要调拨) | **不算**(否则永远不会触发) |

这两个判定使用相同的库存口径但**目的不同**——前者预测未来,后者驱动当下行动。

### 2.7.5 店铺级数据与海外仓(决策 23,经二次简化)

**背景**:同一区域(美国 / 加拿大 / 德国-欧洲共用 / 日本各自)内,任何店铺账号卖同型号,都能从该区域海外仓统一调拨该型号库存。但 **FBA 库存按店铺账号隔离、不能跨号互发**,跨号唯一通道是海外仓。ERP 按两层组织数据(对应界面「合计 / 店铺切换」):

**第一层「合计」(区域汇总)—— 下单 / 区域三态预警在这层算**
- 销量、FBA 可售、海外仓池、在途 = 该区域该型号**所有店铺账号聚合**
- 海外仓共享池显示**总量**;本期下单(决策 10)的"海外仓默认在途" = 整个池
- 区域三态预警(绿 / 黄 / 红 断货救援)在这层判
- ⚠️ **已知局限(接受,见下方设计取舍)**:聚合 FBA 含各店库存,某店 FBA 过剩(滞销囤货)时会抵掉缺货店的下单需求(Codex 新1)。因"同款可切店销售"缓解(缺货店可切到有货店卖),暂接受合计层聚合;待后续做完整店铺维度时再按各店正缺口精算。

**第二层「店铺」—— 只做主力店库存健康预警,不做店铺级断货精算 / 分摊**

> ⚠️ **2026-05-31 二次简化(设计取舍)**:店铺级"断货精算预警 + 按需分摊"**全部砍掉**。原因:① **同款产品 A 店没货可直接切 B 店卖**,店铺级断货不致命,不值得为它做复杂的 ETA 精算 + 分摊;② 真实痛点不是"店铺断货卖不了",而是"**主力店要做活动时 FBA 库存不够、活动做不了**";③ 完整店铺维度后面按实际需要再补。代价:切店有运营损耗(排名 / review),作为已知代价接受。
>
> **演进留痕(四阶段)**:① 占比 × 虚拟分摊 → ② 按需精算、直接 = 指令 → ③ 报店铺级断货预警 + 人工分配 → ④ 现行:只留主力店库存健康预警,店铺级断货交"切店"人工解决。

- **主力店库存健康预警(为活动保障)**:对**配置指定的主力店**(默认大号 1、大号 2)
  ```
  该主力店 FBA 可售天数 = 该店 FBA 可售 ÷ 该店日销
  ```
  < **15 天**(可配置参数)→ 亮"主力店库存预警",推销售 / 采购部。目的 = 保证主力店随时有货做活动,**不是**防断货(断货可切店)。**只看 FBA 可售、不掺在途 ETA**(保持简单,躲开"在途何时到"的精算)。
- **处理**:人工补货——优先从海外仓池调拨给该主力店(走海外仓→FBA 合法通道),或催改派 / 下单。
- **非主力店**:不做店铺级库存预警;断货时由销售**切到同款有货店铺销售**解决。

**海外仓调拨分配**:从哪个店 / 池匀、发多少,由**销售 / 采购部人工定**;系统提供参考(区域池余量 + 主力店库存预警 + 各店 FBA 可售天数)。**系统给信号,不替人拍店铺分配。**

**数据基础**:店铺级 FBA 可售 / 日销来自 ERP **账号维度明细**。

> **待细化**(后议):海外仓发货层(模拟发货单缺"海外仓发货"选项、检测品紧急时优先用海外仓库存、新旧版本备注);周内调拨重分配;欧洲共用德国仓时效差异;海外仓→FBA 调拨量的**箱规 / MOQ 取整**;**完整店铺维度下单**(各店正缺口精算,解 Codex 新1 过剩 FBA 抵扣 + 新5 改派挖空主力店)留后续按实际补;主力店清单维护。

### 2.7.6 跨站未交货订单改派(决策 24)

**两条默认规则(先分清层级)**:

| 层面 | 规则 |
|---|---|
| **海外仓(成品库存)** | **跨区域所有站点之间默认不互调**(美/加/德-欧洲/日 各区域仓地理远,成品不跨区搬)。⚠️ 与 2.7.5(决策 23)不冲突:决策 23 是**同区域内跨账号**共享同一个区域海外仓池;这里说的是**跨区域**(如美国仓↔日本仓)不互调。两个层级。 |
| **国内工厂端(未交货订单)** | **所有站点之间默认可互相改派**(改订单的目的地站点)。系统**不预判规格通用性**——规格 / 包装 / 说明书能不能用,由**采购部按情况人工判断处理**,系统不卡这道闸。 |

**本质**:不是搬库存,是在工厂端改"未交货订单"的目的地。A 站缺货要下单时,若 B 站有"滞销 / 超储 + 未交货的富余订单",系统建议把 B 站这批订单改派给 A 站,A 站本期少下单——既省 A 站重复下单,又消化 B 站滞销在产库存。

**触发逻辑(三条件 AND)**:
1. **A 站(缺货方)**:本期覆盖缺口 > 0,按决策 10 公式要下单
2. **B 站(余货方)**:同型号、有**未交货**订单余量,且 B 站处于滞销 / 超储(这批订单到了对 B 也是积压)。**排除态(2026-06-10 拍板,Q2)**:**B 站处于断货中 / 恢复期不得作为让单方**——其"滞销"大概率是库销比虚高误标(2.5.6 排除态 + 2.2.2 根因),P6 兜不住(覆盖期需求用被污染的预估算出"不差货"),让单雪上加霜。**淘汰期 SKU 的未交货订单自动进改派扫描、让单优先级最高(Q3)**——停备货的在产订单是最纯粹该让的单。
3. → 系统出「跨站改派建议」:B 站未交货订单调 X 件目的地到 A 站,A 站本期下单减 X

**改派数量**:
```
B站可释放余量 = B站未交货订单 − max(0, B站覆盖期需求 − B站现有可用库存)
              (B站滞销时后半段通常≈0,几乎全可释放)
实际改派量   = min(A站本期应下单量, B站可释放余量)
A站剩余缺口  = A站本期应下单量 − 实际改派量   → 这部分A站正常下单
```
⚠️ **「B 站现有可用库存」口径(P6,致命点)**:= B 站**已在本地的现货 + 已锁定不可改派的在途**,**必须排除本次待判断的未交货订单**。若误把那批未交货订单算进"现有可用库存",B 站会被公式判成"库存已够",可释放余量虚高、订单被全改派走,B 站直接断货。
先扣掉 B 站自己未来真还差的量,余下才放——**防 B 站误判滞销、订单全改走后反而断货**。

**三个衔接点(防踩认知坑)**:
- **改派 ≠ 取消订单**(衔接 2.9.1 决策 19):工厂订单总量不变,只改收货站点,**不触发"不可取消"红线**。改派比取消温和,这正是它的价值。
- **改派不是救急**(衔接认知坑③下单/发货/三态预警职责不混):A 站拿到的货 = B 站订单当前进度 + 剩余生产 + 改派到 A 的海运时效,要按改派后路径**重算 A 站三态预警的预计到货**。**若 A 站红预警(必断),改派来的货到不了,救急仍归空派 / 急单**;改派只省"常规重复下单 + 消化 B 站滞销在产"。
- **谁拍板(2026-06-10 审核降级,实质变更)**:改派**默认执行、无审批流**——系统出建议后,仅采购部一道**"规格可行性放行"**(美/日/欧规、包装、说明书;**产线已进定制工序则不可改**),可行即过、留痕。~~原"采购部审核 + 跟两站销售确认 B 站确实释放"已砍~~。
  - **规格放行去留(2026-06-10 拍板,Q6)**:**保留,首版不砍**——它是改派链上唯一人工触点,B 站主力店兜底(Codex 新5)寄生于此;规格通用程度未实证,先跑数据。若后续证实各站规格全通用再砍,**砍时须同步为主力店兜底找新落点**。
  - **执行后自动抄送(2026-06-10 拍板,D4 修复)**:规格放行通过即执行,系统**自动抄送两站销售 + 两站主管**(知会、非确认,不设回执卡点),抄送内容:改派量 / 对 B 站水位影响(改派后 B 站预计断货日变化)/ **异议入口**(B 站销售认为误判 → 走异议通道,不阻塞已执行改派、供下轮校正)。改派动态同步进主管团队聚合视图(决策 13 批注①)。留痕 ≠ 知情,抄送补上两站销售确认砍掉后的知情缺口。
- **改派确认后原子化双边重算**(P7,库存算法一致性、非财务):采购部确认改派 X 件后,系统必须**同步双边**——A 站增加"改派在途 X"、B 站未交货订单减 X,**两边的覆盖期 / 区域三态预警 / 下次下单量一起重算**。否则只在 A 侧扣下单、B 侧仍显示原未交货量,下轮 B 误以为有那批在途 → B 也少下单 → 后面断货。
- **改派在途落"区域层"、不落具体店铺(P7 + Codex 新4)**:改派来的 X 件计入 A 站所属**区域的"改派在途 / 区域可调资源"(合计层)**,**不直接计入任何具体店铺的在途**;落到哪个店铺由后续**人工调拨分配**决定(衔接 2.7.5 二次简化:店铺分配交人工)。避免"加合计层 → 店铺仍红灯 / 加店铺层 → 系统替人做了分配"两难。

> **待细化**(后议,留实现 / 全文重整时定):
> - ~~改派量是否要看 A 站拿货时效(改派来的货到 A 站太晚就不算"省下单",要回退正常下单)~~ **(2026-06-10 裁决:到货时效不影响改派决定)**——改派是公司库存周转率最优解(B 站未交货订单 = 已占用资金,A 站有真需求即优先消化);A 站若等不及,救援归空派 / 急单,两条线不冲突。**"预计到 A 站"仅作到货预告 + 救援补位提示**,不作判断条件。
> - B 站"未交货订单"的口径(只含未发船,还是含已在途;在途改派转运成本由采购部评估,系统默认只建议未发船段)
> - 多个 B 站可同时给 A 站时的优先级(哪个 B 站先释放)
> - 改派后两站账面订单 / 采购单的归属调整(财务 / 采购单据怎么改)
> - **B 站可释放余量是否要扣 B 站内部主力店缺口**(Codex 新5):当前用 B 站合计口径,可能把 B 站某主力店的"救命单"改走;留后续做店铺维度时精算,暂由**采购部规格放行时人工核对 B 站主力店 15 天库存预警**兜底(⚠ 注意:审核降级后这道兜底只剩规格放行一个触点,若规格放行也砍则兜底落空,与上方开口①联动)

### 2.9.1 工厂订单不可取消的边界(决策 19)

**单一边界:终审通过**

```
销售提交 → 采购部审核 → 终审 → 终审通过 = 下单不可取消
                                  ↓
                          采购部发合同给工厂,工厂开始排产
```

终审通过后,采购部一定会把订单发给工厂(合同已下),所以从公司角度看,**终审通过就是不可取消的边界**。

### 2.9.2 五种"锁定"的统一矩阵(防混淆;2026-06-10 由三种扩为五种,K3 诊断落档)

下单不可取消、W1/W2 锁定、项目降级反悔、链接异常冻结、非下单周锁定是五种不同的锁定,边界、作用对象、突破通道都不同。**本表是锁定语义的单一口径源**,各处只引本编号:

| 锁定 | 边界 | 锁什么 / 锁谁 | 谁 / 什么能突破 | 留痕 |
|---|---|---|---|---|
| **下单不可取消** | 终审通过后 | 该笔订单的总量 / 全员 | 无(跨站改派只改目的地、不改总量,2.7.6,不算突破) | 合同 + 系统 |
| **W1/W2 锁定** | 工厂排产 / 备料 | ~~该周的发货量 / 销售~~ <font color="red">仅锁「销售推迟 / 双角色减量」(守工厂节奏,2026-07-07)</font> | 黄 / 红预警流程(2.4.5:绿拦截、黄红走预警两阶段);<font color="red">人工通道:销售提前 / 加量 / 改渠道走主管链,采购推迟 / 提前 / 加量 / 改渠道直改留痕(2.4.5 矩阵)</font>;链接异常冻结(2.4.10) | 系统自动 |
| **项目降级反悔** | 还没启动后续下单 | 后续未下的订单 / — | 未启动的可停(详见 2.11.6) | 系统 |
| **链接异常冻结**(2.4.10) | 链接下架 / 暂停 / 审核中 | 全部未发货计划 + 库存 / 系统自动、全员 | 链接状态恢复;**优先级高于 W1/W2 锁定**;动作层全停见 2.5.8 | 即时通知销售 + 采购部 |
| **非下单周锁定**(2026-06-10 新增,原型节奏条) | 双周下单周期的非下单周 | 本期下单建议(只读)/ 销售 | **紧急追加**:救援通道常开、不受双周节奏限制,入口跳断货特殊处理,审批同断货救援链路(销售 → 主管 → 采购部 → 终审,2.9 矩阵) | 申请 + 审批全程 |

**关键区分**:
- **下单** = 给工厂下采购单(决定生产多少);**发货** = 工厂出货到 FBA(决定每周出多少)。"下单不可取消"管订单总量,"W1/W2 锁定"管单周发货量,"非下单周锁定"管下单节奏,不能混。
- 项目降级反悔与新品 3 次下单 buffer 的关系见 2.11.6。多锁定与多预警共存时的动作取舍见 **2.5.8 仲裁矩阵**。

## 2.11 新品首批备货字段

> **决策来源**:决策 17。新品首批备货逻辑与老品相反,字段独立于 2.5 老品算法链。

### 2.11.1 设计原则

**新品逻辑与老品相反**:
- 老品:先估销量 → 算下单量 → 再发货
- 新品:**先一次性下足总量(覆盖整个新品爬坡期)→ 用发货分批吸收销量不确定性**

**总下单量 = 总发货量**(所有货都发到 FBA,无独立应急储备)。

### 2.11.2 销售填写字段(4 个客观字段)

| 字段 | 含义 | 来源 |
|---|---|---|
| **出货日期** | 项目方告知的出货日期 | 销售填 |
| **目标销售站点** | 新品首发站点 | 销售填 |
| **对标老品** | 必填。系统据此抓取对标老品的月度销量历史 | 销售填 |
| **产品等级** | S / A / B(无 C 级)。从产品主数据自动拽,销售只需确认 | 自动 + 销售确认 |

### 2.11.3 系数字段

| 字段 | 取值 | 说明 |
|---|---|---|
| **产品等级系数** | S = 1.2 / A = 1.0 / B = 0.8 | 仅新品首批用 |
| **销售推新系数** | 主管 / 高级 = 1.0,中级 = 0.9,初级 = 0.8 | 只减不加(保护性下调) |

### 2.11.4 新品首批总量公式

```
新品首批总量 = 对标老品(第 1 月 + 第 2 月 + 第 3 月销量)× 销售推新系数 × 产品等级系数
```

总下单量 = 总发货量(无独立应急储备)。

### 2.11.5 发货分批字段(3 批,3-2-1 递减)

| 字段 | 数量 | 到货时间 |
|---|---|---|
| **发货 1** | 总量 ×(对标第 3 月销量 ÷ 对标三月总销量) | 开售前到 FBA |
| **发货 2** | 总量 ×(对标第 2 月销量 ÷ 对标三月总销量) | 开售第 30 天到 FBA |
| **发货 3** | 总量 ×(对标第 1 月销量 ÷ 对标三月总销量) | 开售第 60 天到 FBA |

按对标老品销量爬坡顺序递减发货(第 3 月最大、第 1 月最小),贴合新品上市后的销量爬坡曲线。

### 2.11.6 下单分批字段(下单 3 次,对应 3 批发货)

**新品首批是下单 3 次,不是 2 次**。每批发货反推工厂下单日:

```
工厂下单日 = 该批发货到 FBA 日期 − 入仓时间 − 快船时效 − 产品交期
```

| 字段 | 下单时点 | 对应发货 |
|---|---|---|
| **第一次下单** | 工厂下单日 = 今天 | 发货 1 |
| **第二次下单** | 约第一次 + 30 天 | 发货 2 |
| **第三次下单** | 约第二次 + 30 天 | 发货 3 |

**项目降级 buffer(天然形成)**:每次下单间隔约 30 天。任意时刻发现项目降级 → 不启动后续下单 → 自动减少压货风险。每次下单提交终审通过后即锁定该次订单,但未启动的第二次 / 第三次下单仍可因项目降级停止——这是新品 3 次下单天然形成的压货风险缓冲。

> 订单不可取消的边界与三种锁定的完整定义见 2.9(属决策 19),此处只引用,不展开。

### 2.11.7 运输方式

**默认走快船**(新品专用,比海运快约 7 天,缩短提前期;突发爆发时可空派补救)。

### 2.11.8 开售后并入老品算法(决策 20)

新品**没有独立的过渡机制**:开售当天起即并入老品算法体系,不设"开售+14 天切换"等时间切换点。

- **开售当天**:SKU 进入库存水位表,实际销量从第 1 天起统计,直接喂本期下单 / 发货 / 三态预警 / 顺延机制(与老品同一套)。
- **下单接管靠"覆盖缺口"驱动,不看日历**:沿用老品~~本期下单算法(决策 10 修订…)~~<font color="red">发货计划逐周滚动 + 净额链路(2.5.2 修订,2026-07-07):发货计划排出缺口周,下单批合计减未交货余量出净额;未交货订单只在净额抵扣</font>。缺口出现的早晚自然对应销量快慢。
- **早期销量不稳不影响**:开售前两批货(发货 1 开售即到、发货 2 第 30 天到)库存充足,为实际销量提供"攒稳样本"的时间窗口;开售初期的测评 / 广告虚火,因库存足、滚动模拟短期不出缺口,不会驱动误下单。
- **三情景全由老品现有机制接住**:
  - 卖平 / 卖慢 → 缺口晚现,数据稳后本期下单接管;货发多了则发货 3 走顺延机制推迟(纯延时,默认整批,系统定整批或部分)。
  - 卖爆 → 三态预警捕捉,空派 / 工厂急单。
- **运输方式**:每笔下单按三态预警自然判定(够海运→海运、不够→空派、必断→急单),不设"第几笔切海运"硬规则。
- **对标老品的作用边界**:仅用于开售前首批总量估算(2.11.4),开售当天起退场,后续全部用实际销量。

> 决策 20 收口:取消原"开售+14 天切换"设计。新品的特殊逻辑收缩到只剩开售前的首批下单(2.11.1–2.11.7);开售后零特殊,完全并入老品。"开售+14 天"若保留,仅作销售人工动销复盘节点,不进算法。

### 2.11.9 新品首批销售申请校准(2026-06-02 拍板)

**背景**:新品零历史销量,2.11.4 的系统测算量是基于对标老品的**客观基准**。销售若要偏离这个基准,属于"用主观判断覆盖客观算法",在新品高风险场景下必须留下依据,不设任何"小幅免理由"的灰色地带。

**规则**:
- **系统测算量 = 客观锚点**;销售可在申请单上填"销售申请量"对其校准。
- **新品首批 100% 走审核**(销售 → 采购部 → 终审,见 2.9),**不设免审门槛**。
- **只要"销售申请量 ≠ 系统测算量"(任何幅度的上调或下调),即强制填写「首月促销计划 + 文字理由」方可提交**;**不设偏离比例门槛**(区别于老品下单异议 2.9.4 的 ±30% 阈值——新品但凡调整就要理由,老品是超阈值才要)。
- **销售申请量 = 系统测算量**(完全采纳系统建议)→ 无需额外理由,直接进审批流。

**与 2.9.4 的区分**:2.9.4 是老品本期下单的"异议管理"(超 ±30% 才强制理由);本节是新品首批的"申请校准"(但凡偏离就强制理由)。两者语境、阈值都不同,各自独立,不共用编号。

**留痕**:申请量、系统测算量、偏差、促销计划、理由、操作人、日期一并留档,供采购部/终审审议及事后复盘。
