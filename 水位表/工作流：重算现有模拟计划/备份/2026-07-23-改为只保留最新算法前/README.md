# 工作流：发货计划演变-v2-重算现有建议

## 业务目的

对已经存在的 W1-W7 模拟计划重新计算建议数量，并把缺货期间未满足的需求补回。重算只更新原记录的 `number`，不新增、不删除模拟计划，不改变原计划 ID、发货日期、入仓日期、渠道、季节或入仓天数。

## 文件顺序

1. `01-预览重算结果.sql`：只读预览全部旧值、新值和逐周计算过程。
2. `02-重算现有建议.sql`：生产写回节点，原地更新现有计划并保存最近一次回滚快照。
3. `03-回滚最近一次重算.sql`：从快照恢复最近一次重算前数量，并标记回滚时间。
4. `04-重算后水位推演.sql`：与“更新水位表”节点 10 相同的全量 V2 水位推演，负责把新建议同步到 `daily_sales.v2_*`。

## 计算公式

每个 ASIN、站点、合计行按 W1 到 W7 顺序计算：

```text
覆盖售卖期 = add_date + 7 天 至 add_date + 13 天
14 天目标库存 = 覆盖售卖期 7 天需求 × 2
有效起点库存 = 排除旧 V2 计划后的真实基线库存 + 前序新建议累计
本周新建议 = MAX(0, 14 天目标库存 - 有效起点库存)
```

基线库存按连续欠货口径计算，库存可以为负；补货到达时不会清除之前缺货期间的需求。

## 保护条件

- 只处理 `plan_source = 'shipment_plan_v2'`、`shop = '合计'`、发货日不早于今天的计划。
- 每组必须正好有 7 条不同发货日的计划。
- 任一条计划已关联真实发货，整组跳过。
- 任一条计划存在待审核、待应用、应用中或已应用申请，整组跳过。
- 覆盖售卖期必须有完整 7 天需求数据。
- 线上工作流默认停用；只有人工确认预览后才允许执行。

## 回滚

写回前的 `number` 保存在 `v2_calculation_snapshot.recalculation.previous_number`。回滚节点只恢复最近一次重算，并保留原计划 ID 和其他业务字段。回滚后仍需再次执行 `04-重算后水位推演.sql`。

快照同时写入页面抽屉现有的顶层字段契约，因此不修改 JS 也能展示服务期、14 天需求、有效起点库存、需求缺口和最终建议；完整的基线库存与前序新建议仍保存在 `recalculation` 明细中。
